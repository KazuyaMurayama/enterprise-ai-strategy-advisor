# Enterprise AI Strategy Advisor

## プロジェクト概要
企業名または業界名を入力するだけで、MBB（マッキンゼー・ベイン・BCG）クラスの生成AI戦略レポートを自動生成するシステム。

## FILE_INDEX.md 参照ルール

**`FILE_INDEX.md`** は以下のタイミングで必ず参照すること：

1. **セッション開始時** — ファイル構成・ブランチ状態・生成済みレポートの概要を把握する
2. **これまでの経緯・最新状態を確認すべき時** — バージョン差分・レポート更新履歴を確認する
3. **ファイル参照指示があった時** — 該当ファイルのパス・最新版を特定してから作業する

---

## 品質基準
- **マッキンゼー水準**: MECE構造化、定量インパクト試算、So What?の明確化
- **ベイン水準**: 実行可能性重視の具体的アクションプラン、ROI試算、Full Potential分析
- **BCG水準**: フレームワーク活用（バリューチェーン分析、成熟度モデル、アンビション・マトリクス）
- 数値根拠のない主張は禁止。推定でもロジックと前提を明示（フェルミ推定可）

## AIセキュリティ・ガバナンス基準（全レポート共通）

全てのAI戦略レポートは、以下の10領域を必須でカバーすること。Phase 4（技術アーキテクチャ）とPhase 8（リスク管理）に分散配置する。

### セキュリティ投資ガイドライン
- AI総投資額の **7-10%** をセキュリティ・ガバナンスに配分（業界ベストプラクティス）
- Phase 6の投資計画に「AIセキュリティ投資」を独立項目として計上
- 年間運用コストの15-20%をセキュリティ監視・監査に充当

## レポート生成ワークフロー

### フェーズ構成
1. エグゼクティブサマリー（経営層向け1枚要約）
2. 業界/企業の現状分析（PESTEL、Porter's 5 Forces、競合ベンチマーク）
3. バリューチェーン全体のAI機会マッピング（プライオリティマトリクス、Top10ユースケース）
4. 技術アーキテクチャ戦略（Build/Buy/Partner、基盤モデル選定、データ戦略）
5. 組織・人材戦略（組織設計、人材戦略、チェンジマネジメント）
6. 財務モデルとROI試算（投資計画、リターン試算、感度分析）
7. 実行ロードマップ（Phase 0-4、36ヶ月計画）
8. リスク管理（リスク一覧、AI倫理・ガバナンス）

### 各フェーズの必須ルール
- 各フェーズ冒頭に「このセクションの要点」を3行で記載
- すべての提言に [推奨] タグと理由を付ける
- 数値のない提言は禁止
- 表・マトリクス・フレームワークを多用して視認性確保
- MECE構造化を徹底

## ファイル保存規則
- レポートは `output/{企業名or業界名}/` 配下に保存
- 各フェーズは個別ファイルとして保存: `phase{N}_{section_name}.md`
- 目次ファイルを `output/{name}/README.md` として作成
- 最終統合レポートを `output/{name}/full_report.md` として作成

## カスタムスキル（スラッシュコマンド）

### /ai-strategy-full（★推奨: 統合コマンド）
一気通貫コマンド。レポート生成→品質改善→クライアントレビュー→PPTX生成まで全自動。
使用例: `/ai-strategy-full トヨタ自動車`
使用例: `/ai-strategy-full 日本の金融業界 --skip-pptx`
オプション: `--skip-review`（レビュースキップ）、`--skip-pptx`（PPTXスキップ）、`--scale small|medium|large`

### /ai-strategy
レポート生成のみ（Stage 1）。企業名/業界名を引数に取り、フェーズ1-8の全レポートを順次生成。
使用例: `/ai-strategy トヨタ自動車`

### /industry-analysis
フェーズ2の業界・企業分析のみを実行。PESTEL、Porter's 5 Forces、競合ベンチマーク。
使用例: `/industry-analysis 製造業`

### /usecase-mapper
フェーズ3のバリューチェーンAI機会マッピングのみを実行。
使用例: `/usecase-mapper 金融業界`

### /roi-calculator
フェーズ6の財務モデル・ROI試算のみを実行。
使用例: `/roi-calculator 従業員5000名 売上高1000億円`

### /executive-summary
フェーズ1のエグゼクティブサマリーのみを生成。既存分析結果がある場合はそれを参照。
使用例: `/executive-summary トヨタ自動車`

## エージェントチーム設計

レポート生成を並列化するため、以下のエージェントチームを使用:

1. **Research Agent（調査エージェント）**: WebSearch/WebFetchで業界・企業情報を収集
2. **Strategy Agent（戦略エージェント）**: 収集情報を基にMBB水準の戦略分析を実施
3. **Financial Agent（財務エージェント）**: ROI試算、感度分析、財務モデル構築
4. **Technical Agent（技術エージェント）**: 技術アーキテクチャ、モデル選定、データ戦略
5. **Report Agent（レポートエージェント）**: 全フェーズの統合・品質チェック・最終化

## 🚀 自動実行ポリシー

### 確認不要で即実行する操作
- 全ファイル操作（作成・編集・削除）
- 全シェルコマンド（PowerShell, Bash, git, gh, npm, node, python）
- Git操作: add / commit / push（featureブランチ）/ pull / fetch / merge / branch -D / reset --hard
- GitHub操作: gh pr create / gh api 全般 / ブランチ削除
- パッケージ操作: npm install / pip install
- Web検索・フェッチ
- バックグラウンドプロセス起動

### 事前確認が必要な操作（例外のみ）
- `git push --force` を main / master ブランチに対して実行する場合
- `gh repo delete` 実行時

### 動作原則
- 計画提示（簡潔）→ 即実行 → 結果報告 のフロー厳守
- 事前確認文（「Should I run...?」等）を出力しない
- エラー時は即再試行 or 別アプローチで対応、判断が必要な場合のみ報告

- 更新時は **最終更新日のみ** を当日付に書き換える（作成日は固定）
- 除外: README / CLAUDE.md / FILE_INDEX / tasks.md / CHANGELOG / LICENSE

## 作業品質ルール

### Git・ブランチ管理
- 作業前: `git branch --show-current` でブランチ確認 → main以外なら `git checkout main && git pull` してから開始。

### ファイル特定（編集前）
- ユーザー発話のキーワード全てをファイル名と照合してから編集。キーワード不完全一致・候補不確かなら必ず確認。

### 成果物報告
- ファイル作成・更新・push後は必ず3列表で報告: `| 成果物 | 説明 | リンク |`
- リンクは `/blob/<実ブランチ>/<パス>` 形式。報告前に `gh api repos/OWNER/REPO/contents/PATH?ref=BRANCH` で存在確認。push前はURL生成しない。

### ドキュメント品質
- UIパス・コマンド・設定名は公式ドキュメントで確認後に記載。確認不可なら「[要確認]」と明記。
- OS/環境制約（例: Windows専用）をタスク開始時に確認。完成後に `brew`/`Cmd`/`macOS` 等をgrepして除去。
