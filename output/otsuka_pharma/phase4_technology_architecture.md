# フェーズ4: 技術アーキテクチャ戦略 — 大塚製薬株式会社 医薬品事業

> 作成日: 2026年4月14日

---

> **このセクションの要点**
> - 大塚製薬のAIアーキテクチャはGxP規制強度に応じた3層設計が必須。Layer A（非GxP業務AI）・Layer B（GxP対象：臨床・製造AI）・Layer C（GxP最高強度：創薬AIデータプラットフォーム）に分離し、計算機化システムバリデーション（CSV）をLayer B/Cに適用
> - 最大の技術課題は「分散する研究・臨床・製造データの統合」。各部門のデータサイロ（ELN・EDC・LIMS・MES）をAIデータプラットフォームで統合することが全AI施策の前提条件
> - 基盤モデルはMR支援・業務AIにClaude Sonnet 4.6、薬事文書生成にClaude Opus 4.6、創薬AIには専門プラットフォーム（Schrödinger・Recursion等）とのPartner戦略が最適

## 4-1. 製薬AI 3層アーキテクチャ（大塚製薬版）

```
│  Layer B: GxP対象AI（CSV/バリデーション必須）                      │
│  [臨床試験設計AI] [患者適格スクリーニングAI] [安全性シグナルAI]    │
│  [GMP品質予測AI] [製造逸脱AI] [RWE解析AI]                       │
│  ↑ バリデーション済みAIプラットフォーム（GAMP5準拠）              │
│  ↑ ICH E9(R1) / PMDA AIガイドライン準拠                         │
├────────────────────────────────────────────────────────────────┤
│  Layer C: 創薬AIデータプラットフォーム（GxP最高強度）               │
│  [CNS創薬AI（標的探索）] [ADMET予測AI] [Taiho Oncology AI]      │
│  [分子生成AI] [DTx×AI統合]                                      │
│  ↑ 専門創薬AIプラットフォーム（Schrödinger, Recursion等）         │

└────────────────────────────────────────────────────────────────┘
              ↕  製薬AIデータ統合プラットフォーム
              ↕  ELN / EDC / LIMS / MES / CRM データブリッジ
              ↕  Azure Private Link + Zero Trust基盤
```

## 4-3. Build / Buy / Partner 意思決定

| ユースケース | 戦略 | 理由 | 主な選択肢 |
|-----------|:----:|-----|:--------:|
| MR支援AIアシスタント | **Buy + カスタマイズ** | 汎用LLM + 大塚固有薬事RAGで差別化 | Claude API + Azure Search |
| 薬事CTD生成AI | **Buy + カスタマイズ** | 薬事RAG精度が核心。Claude Opus品質 | Claude Opus API + 薬事DB |
| 社内ナレッジ管理 | **Buy（クラウドAPI）** | RAGフレームワーク活用 | Azure OpenAI + Cognitive Search |
| CNS創薬AI | **Partner（専門外部）** | AlphaFold統合・分子設計は専門プラットフォームが不可欠 | Schrödinger, Recursion Pharmaceuticals |
| ADMET予測AI | **Buy（SaaS）** | 学習済み高品質モデルが市場に存在 | Schrödinger ADME, Optibrium StarDrop |
| GMP品質予測AI | **Build（in-house）** | 大塚固有製造プロセスデータが差別化源泉。外部ベンダーには渡せない | PyTorch + Azure ML |
| AI臨床試験設計 | **Partner（専門外部）** | ICH準拠の適応型試験設計AIはCovance・PharmaLexが強み | Medidata AI, Veeva Vault |
| 安全性シグナルAI | **Buy（GxP認証済み）** | PMDA/FDA対応済みシステムが市場に存在 | Oracle Argus AI, IQVIA Safety AI |

## 4-5. GxPバリデーション戦略（GAMP5準拠）

```
【AIシステムのGxP分類判定】
       ↓
  Layer A（非GxP）: バリデーション不要。社内ガイドライン遵守のみ
       ↓
  Layer B（GxP対象）: GAMP5 Category 4-5として取り扱い
       ↓
  【CSV実施ステップ】
  1. URS（ユーザー要求仕様書）策定
  2. リスクアセスメント（機能リスク × GxP影響度）
  3. DQ（設計時適格性確認）
  4. IQ（据付時適格性確認）
  5. OQ（運転時適格性確認）
  6. PQ（性能適格性確認）
  7. 変更管理（AIモデルバージョンアップ時の再バリデーション）
  8. 定期的なシステムレビュー（年1回）
       ↓
  Layer C（GxP最高強度）: + GLP電子記録・電子署名（21 CFR Part 11）
```

## 4-6. AIセキュリティ10領域フレームワーク（Phase 4担当：領域2・5・7・9）

### 領域2：AIサプライチェーンセキュリティ

| 対策項目 | 大塚製薬への適用 | 優先度 |
|---------|-------------|:-----:|
| **外部LLMサービスの認証確認** | Claude/Azure OpenAI利用前にSOC2 Type II・GxP SaaS条項確認。年次更新 | ★★★★★ |
| **創薬AIプラットフォームの検証** | Schrödinger・Recursion等の専門ベンダーのGxP適合性確認（CSV対象かを判定） | ★★★★★ |
| **OSS依存関係スキャン** | MR支援AIのOSSライブラリにSnyk適用。薬事情報漏洩リスクの依存関係を監視 | ★★★★☆ |
| **モデルファイルのハッシュ検証** | GxP対象AIの本番デプロイ時にSHA-256検証。改ざん防止をCSV手順書に明記 | ★★★★★ |

### 領域5：プライバシー保護技術（PETs）

| 技術 | 製薬業での適用 | 実装方針 |
|-----|------------|---------|
| **差分プライバシー** | 臨床試験患者データをAI学習に使う際の個人識別防止。ε=1.0以下を推奨 | Layer B/C AIモデルの学習段階に適用 |
| **連合学習** | 医療機関・治験施設の患者データを外に出さずにAI学習（グローバル試験） | DTx連携・多施設バイオマーカーAIで検討 |
| **仮名化・匿名化** | EDCの患者識別子をトークン化。AIシステムは仮名ID（Subject ID）のみ参照 | 全Layer B/CのAI入力データに適用 |
| **合成データ生成** | 希少疾患（患者数少）の学習データ不足を合成データで補完 | CNS創薬AIの前臨床データ拡張に活用 |

### 領域7：データリネージ

| 管理対象 | リネージ追跡要件 | ツール |
|---------|------------|:----:|
| **創薬AI学習データ** | ELN化合物データ → クリーニング → 特徴量変換 → 学習データまでの全ステップ（GLP対応） | Apache Atlas + MLflow |
| **臨床試験AI入力** | EDCデータ → CDISC変換 → AI入力までの完全リネージ（PMDA/FDA監査対応） | Veeva Vault + カスタムDB |
| **GMP品質AIの入力データ** | MESセンサーデータ → 前処理 → AI推論までのGxPリネージ記録 | OSIsoft PI + Azure ML |
| **LLM参照文書（RAG）** | MR支援AI・薬事AIが参照した添付文書・社内SOP・学術論文のバージョン記録 | LangSmith |

### 領域9：Zero Trust for AI

| 原則 | 大塚製薬AIへの適用 | 実装 |
|-----|----------------|-----|
| **最小権限** | Layer A（MR支援AI）とLayer C（創薬AIデータ）のアクセス権を完全分離。研究者のみLayer Cにアクセス可 | Azure RBAC + API Management |
| **継続的認証** | 研究者の創薬AIアクセスはMFA必須。長時間セッションは自動タイムアウト（30分） | Azure AD + 条件付きアクセス |
| **マイクロセグメンテーション** | 患者データ（EDC）へのアクセスはLayer B専用ネットワーク経由のみ。Layer Aからの直接参照を禁止 | Azure VNet + NSG |
| **AI入出力検査** | 全AIへの入出力をスキャン。未公表化合物情報・患者個人情報の外部LLMへの意図しない送信を自動ブロック | Azure AI Content Safety + DLP |
