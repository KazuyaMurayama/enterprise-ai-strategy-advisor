# フェーズ4: 技術アーキテクチャ戦略 — 大樹生命保険株式会社

> 作成日: 2026年4月13日

---

> **このセクションの要点**
> - 大樹生命のAIアーキテクチャは「金融庁規制対応必須の引受・支払AI領域」「顧客・外交員向けビジネスAI領域」「社内業務AI領域」の3層分離設計が必要。特に引受AIは金融庁との協議完了前の本番稼働は不可
> - 最大の技術課題は「老朽化した勘定系システム（COBOL/メインフレーム）とのAPI連携」。AIシステムがどれだけ優れていても、契約DB・保険金支払DBへの安全なアクセス経路がなければ機能しない
> - 基盤モデルはClaude Sonnet 4.6（日本語対話・文書生成）とAzure OpenAIの組み合わせを主軸に、保険引受特化モデルは外部専門ベンダー（Munich Re Digital Partners等）との提携を推奨

---

## 4-1. 技術アーキテクチャ全体設計

### 大樹生命 保険AI 3層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│  Layer A: 社内業務AI（非規制対象）                                  │
│  [ナレッジ管理AI] [外交員提案書AI] [コールセンターAI] [書類OCR AI]  │
│  [代理店支援AI] [競合分析AI] [財務報告AI] [HR AI]                  │
│  ↑ Claude Sonnet 4.6 API / Azure OpenAI  + RAG基盤               │
├──────────────────────────────────────────────────────────────────┤
│  Layer B: ビジネスコアAI（金融庁届出・監督下）                        │
│  [引受AI（リスク評価補助）] [不正検知AI] [解約予測AI] [資産運用AI]   │
│  ↑ バリデーション済みAIプラットフォーム                             │
│  ↑ 金融庁届出・内部統制文書化必須                                   │
├──────────────────────────────────────────────────────────────────┤
│  Layer C: 保険金支払審査AI（最高規制強度）                           │
│  [支払可否判定補助AI] [医的情報解析AI] [不払い防止AI]               │
│  ↑ 専門保険AIベンダー（金融庁認可済みシステム）                     │
│  ↑ 完全な監査ログ・説明可能性（XAI）必須                            │
└──────────────────────────────────────────────────────────────────┘
           ↕  保険AI中間層API（勘定系との安全なブリッジ）
           ↕  契約DB / 保険金支払DB / 顧客DB / 運用DB
           ↕  セキュリティ・Zero Trust基盤
```

## 4-2. 勘定系レガシーとの統合戦略（大樹生命最大の技術課題）

### 勘定系APIブリッジ設計

| 連携対象システム | 技術的課題 | 対応方針 | 期間・コスト |
|--------------|---------|---------|:----------:|
| 保険契約DB（COBOL/メインフレーム） | APIが存在しない。バッチ処理中心 | API Gatewayによる安全なラッパー構築。読み取り専用APIを優先実装 | 6-9ヶ月 / 1.5億円 |
| 保険金支払DB | 個人医的情報含む。アクセス制御が厳格 | 匿名化・アクセスログ必須のセキュアAPI。Layer C専用エンドポイント | 6-12ヶ月 / 1.0億円 |
| 外交員CRM（SalesForce等） | AI連携APIは一部提供済み | 既存API活用＋AIアシスタントのWebhook統合 | 2-3ヶ月 / 0.3億円 |
| コールセンターCTI | APIなし | CTIベンダーとのカスタム連携開発 | 4-6ヶ月 / 0.5億円 |

## 4-6. 金融庁規制対応AIシステム導入フレームワーク

```
【STEP 1】AI適用の規制分類（保険業法・金融庁ガイドライン上の位置づけを確認）
       ↓
【STEP 2】金融庁への事前相談（非公式）→ 問題がなければ正式届出
       ↓
【STEP 3】内部監査部門・コンプライアンス部門による審査
       ↓
【STEP 4】小規模パイロット（対象を限定したPoC、数百件のみ）
       ↓
【STEP 5】効果測定・公平性検証・XAIレポート作成
       ↓
【STEP 6】金融庁への正式届出（引受AI・支払審査AIの場合）
       ↓
【STEP 7】段階的な本番展開（50% → 75% → 100%の段階拡大）
       ↓
【STEP 8】定期的な公平性監査・モデルバージョン更新の届出管理
```

## 4-8. AIセキュリティ技術実装（10領域フレームワーク：Phase 4担当4領域）

### 領域2：AIサプライチェーンセキュリティ

| 対策項目 | 大樹生命への適用 | 実装優先度 |
|---------|--------------|:--------:|
| **外部LLMサービスの認証確認** | Claude/Azure OpenAI利用前にSOC2 Type II・金融機関向け条項を確認。年次更新義務化 | ★★★★★ |
| **AI APIの依存関係スキャン** | 外交員AIアシスタントで使用するOSSライブラリにSnyk/Dependabotを適用 | ★★★★☆ |
| **モデルファイルのハッシュ検証** | 本番デプロイ時にSHA-256ハッシュ検証。不正改ざんを自動検知・ブロック | ★★★★★ |
| **保険データ処理APIの暗号化** | 契約DB・支払DBへのAPIリクエストはmTLS（相互TLS認証）必須 | ★★★★★ |

---

### 領域5：プライバシー保護技術（PETs）

| 技術 | 保険業での適用場面 | 実装方針 |
|-----|---------------|--------|
| **差分プライバシー** | 保険引受AIの学習データ（医的情報含む）への適用。ε=0.5〜2.0で個人の健康情報が推論されないことを保証 | 引受AIモデルの学習段階に必須適用 |
| **連合学習** | 複数医療機関・提携先データを個人情報の外部共有なしに活用 | 健康増進保険の将来的なデータ連携で検討 |
| **k-匿名化（k≥10）** | 顧客属性データを外交員AIが参照する際の匿名化基準 | CRMデータのAI連携時に必須 |
| **トークン化** | 保険証券番号・個人番号のAIシステムへの直接入力を禁止。トークン化した代替IDを使用 | Layer A/B/C全体に適用 |

### 領域7：データリネージ

| 管理対象 | リネージ追跡要件 | ツール |
|---------|-------------|:-----:|
| **引受AI学習データ** | 告知書データ → クリーニング → 特徴量変換 → 学習データセットまでの全変換ステップ記録 | Apache Atlas |
| **保険金支払審査AI入力データ** | 診断書 → OCR → 構造化 → AI入力までの完全なGxP相当リネージ（金融庁監査対応） | カスタムリネージDB |
| **LLM推論の参照文書** | 外交員AI・コールセンターAIのRAGが参照した約款・社内規程のバージョンを記録 | LangSmith |
| **モデルバージョンとデータの紐付け** | どのモデルが、いつ、どのデータで学習されたかをMLflowで追跡 | MLflow |

---

### 領域9：Zero Trust for AI

| Zero Trust原則 | 保険AIへの適用 | 実装方針 |
|-------------|-------------|--------|
| **最小権限の原則** | Layer A（外交員AI）とLayer C（支払審査AI）のAPIキーを完全分離。Layer Cへは医的情報取扱い資格者のみアクセス可 | Azure RBAC + API Management |
| **継続的認証** | 外交員のAIアシスタント利用は生体認証（顔認証・指紋）による継続確認。長時間セッションの不正利用防止 | Azure Active Directory + 生体認証 |
| **マイクロセグメンテーション** | 契約DB・支払DBへのアクセスはLayer B/Cのみ。Layer Aからの直接DB接続を技術的に禁止 | Azure VNet + NSG + Private Endpoint |
| **AIリクエスト/レスポンス検査** | 全AIシステムへの入出力をリアルタイムスキャン。顧客個人情報・医的情報の意図しない送信を自動ブロック | Azure AI Content Safety + DLP |
