# フェーズ4: 技術アーキテクチャ戦略 — 日本化薬㈱ 医薬品事業

> 作成日: 2026年4月13日

---

> **このセクションの要点**
> - 日本化薬の医薬品AIアーキテクチャは「ADC研究特化レイヤー（社外秘データを一切クラウド送信しない）」「GxP準拠業務レイヤー（バリデーション必須）」「商業・一般業務レイヤー（標準クラウドAI）」の3層分離設計が必須
> - ADCリンカー設計AIのコア知的財産は外部に流出させないため、分子設計計算はプライベートクラウドまたはオンプレミスGPUで実行する原則を貫く
> - 規模が中堅（R&D費100億円）のため、インフラ自社構築は最小限に留め、Schrödinger・Atomwise等の専門プラットフォームとのAPI連携を主軸とするBuy & Partner戦略が最適

## 4-1. 技術アーキテクチャ全体設計

### 日本化薬 医薬品AI 3層アーキテクチャ

```
┌────────────────────────────────────────────────────────────────────┐
│  Layer A: 商業・一般業務AI（非GxP）                                   │
│  [MR支援AI] [ナレッジ管理AI] [MA資料生成AI] [CTD補助] [競合分析AI]    │
│  ↑ 標準クラウドLLM（Claude API / Azure OpenAI）                      │
│  ↑ RAGアーキテクチャ（社内文書＋外部文献）                            │
├────────────────────────────────────────────────────────────────────┤
│  Layer B: GxP準拠業務AI（バリデーション・CSV必須）                     │
│  [CTD自動生成（GxP部分）] [製造品質予測AI] [安全性シグナル検出AI]      │
│  ↑ バリデーション済みAIプラットフォーム（GxP対応環境）                 │
│  ↑ 21 CFR Part 11対応の監査ログ・変更管理                            │
├────────────────────────────────────────────────────────────────────┤
│  Layer C: ADC創薬特化AI（機密データ・プライベート実行必須）             │
│  [リンカー設計AI] [ペイロード毒性予測AI] [DAR最適化AI]                │
│  ↑ オンプレミスGPU または プライベートクラウド（外部送信禁止）          │
│  ↑ Schrödinger API（ローカル実行モード）/ 社内MLモデル               │
└────────────────────────────────────────────────────────────────────┘
              ↕  医薬品AIデータレイク（機密ランク別分離管理）
              ↕  セキュリティ基盤（Zero Trust・DLP・アクセス管理）
```

## 4-2. Build / Buy / Partner 意思決定

### ユースケース別推奨戦略

| ユースケース | 戦略 | 理由 | 主な選択肢・候補 |
|-----------|:----:|-----|:------------:|
| MR支援AIアシスタント | **Buy（クラウドAPI）** | 汎用LLMで十分。RAGカスタマイズのみ自社 | Claude API + Azure Search |
| 社内ナレッジ管理AI | **Buy（クラウドAPI）** | 構築済みRAFフレームワークを活用 | Azure OpenAI + Cognitive Search |
| CTD自動生成 | **Buy + カスタマイズ** | 薬事文書特化RAGが必要。GxP対応ベンダー選定 | Veeva Vault AI, Regulatory AI専門ベンダー |
| ADCリンカー設計AI | **Partner（専門外部）+ 社内微調整** | Schrödinger等の物理シミュレーション必須。社内ADCデータで追加学習 | Schrödinger FEP+, Boltz-2, Atomwise |
| ペイロード毒性予測 | **Partner + Buy** | 毒性予測特化モデルは外部が先行 | pk/pd AI、Lhasa Limited Derek Nexus |
| 製造品質予測AI | **Buy（製造特化）** | GxP対応済みプラットフォームが現実的 | OSIsoft PI + Azure ML, Seeq |
| 患者選択・層別化AI | **Buy（臨床AI）** | 臨床実績のあるプラットフォーム活用 | Medidata Rave AI, Veeva Vault CTMS AI |
| 競合パイプライン分析AI | **Buy（SaaS）** | 専門情報DB付きサービスが安価・高品質 | Citeline, GlobalData, Informa Pharma |

## 4-3. 基盤モデル選定

### 日本化薬 医薬品AI 用途別モデル推奨

| 用途 | 推奨モデル | 選定理由 | 月間コスト試算（400名MR＋200名研究・薬事） |
|-----|:--------:|---------|:-------------------------------------:|
| MR向け日本語AIアシスタント | **Claude Sonnet 4.6（RAG）** | 日本語品質・処理速度・コストバランス最良 | 80-150万円 |
| 文献・特許解析（英語論文） | **Claude Opus 4.6** | 長文科学論文の要約・推論精度が最高水準 | 100-180万円 |
| CTD文書ドラフト生成 | **GPT-4o + ファインチューニング** | 薬事文書形式への高い適応性 | 80-130万円 |
| ADCリンカー分子設計 | **Schrödinger FEP+（ローカル）** | 物理化学シミュレーション統合が必須 | 300-600万円（ライセンス） |
| タンパク質構造予測 | **AlphaFold3（自社サーバー）** | 無償・精度業界標準。ADC接合部位設計に直結 | 10-30万円（インフラのみ） |
| 毒性予測 | **Lhasa Derek Nexus** | 薬事規制対応実績のある専門ツール | 50-100万円（ライセンス） |
| 製造センサーデータ分析 | **Azure Machine Learning（時系列）** | GxP対応環境で時系列AI構築が容易 | 50-100万円 |
| コード・分析支援 | **Claude Sonnet 4.6** | 研究者の分析スクリプト生成・レビュー | 30-60万円 |
| **月間合計** | | | **700-1,350万円** |

> **モデルガバナンス3原則**:
> 1. ADCリンカー設計・分子データは外部LLM APIに送信禁止（プライベート実行必須）
> 2. 臨床試験・患者データは匿名化後のみクラウドAI処理可（re-identification対策を実施）
> 3. GxP対象AIシステムのモデルバージョンアップは変更管理手順書に従いCSV再バリデーション後のみ

## 4-4. データ戦略

### 日本化薬 医薬品データ資産の現状と整備計画

#### データ資産評価

| データ種別 | 推定保有量 | AI活用準備度 | 最大課題 | 優先整備度 |
|----------|:--------:|:----------:|--------|:--------:|
| **ADCリンカーデータ（合成・活性）** | 数万件（推定） | ★★☆☆☆ | 非標準形式・散在。機械学習用フォーマット未整備 | ★★★★★ |
| **製造センサーデータ（高崎・岡山）** | 大量（リアルタイム） | ★★★☆☆ | リアルタイム収集基盤はあるが、AI統合が未実施 | ★★★★☆ |
| **臨床試験データ（EDC）** | Phase 1-3複数試験分 | ★★★☆☆ | GxPデータのAI学習利用の法的整理が必要 | ★★★★☆ |
| **MR・CRMデータ** | 400名分の訪問ログ | ★★★★☆ | 処方データとの統合が不十分 | ★★★★☆ |
| **研究報告書・実験ノート** | 数十年分（紙含む） | ★★☆☆☆ | 非構造化・電子化未完了。AI検索不能 | ★★★★★ |
| **特許・文献データ** | 社内特許+外部文献 | ★★★☆☆ | 社内特許と外部情報の統合基盤なし | ★★★☆☆ |
| **眼科製品臨床・市販後データ** | 国内RWD | ★★★☆☆ | AI解析活用には再構造化が必要 | ★★★☆☆ |

#### データ整備ロードマップ（優先順位順）

**最優先（Month 1-6）: ADCリンカーデータ標準化**
- 化合物構造データのSMILES形式統一
- 活性値（IC50, EC50, stability data）の正規化・タグ付け
- 過去20年分の合成実験データの電子化・DB登録
- **投資**: 0.8-1.2億円

**次優先（Month 3-12）: 研究ノート・報告書のデジタル化**
- 過去5年の研究報告書をOCR＋構造化でデジタル化
- ELN（電子実験ノート）システム導入（将来のナレッジAIの原料）
- **投資**: 0.5-0.8億円

**中期（Month 7-18）: 医薬品AIデータレイク構築**
- Layer A/B/C別のデータ分離管理アーキテクチャ構築
- EDC・CRM・製造システムのデータAPI連携
- 個人情報de-identification自動化基盤
- **投資**: 1.0-1.5億円

## 4-5. GxP準拠AIシステム導入フレームワーク

### 製造品質AI・CTD生成AIへのGxP対応手順

```
【STEP 1】AI適用範囲の規制分類（GxP対象か非GxP対象かの判定）
        ↓
【STEP 2】バリデーション計画（VP）の作成と承認（QA部門）
        ↓
【STEP 3】PMDA事前相談（AI活用計画の規制当局への事前説明）
        ↓
【STEP 4】IQ（導入適格性確認）→ OQ（運用適格性確認）→ PQ（性能適格性確認）
        ↓
【STEP 5】CSV（コンピュータ化システムバリデーション）完了文書作成
        ↓
【STEP 6】本番運用開始 → 定期レビュー（年1回以上）
```

### GxP対応コスト概算（日本化薬規模）

| 対応項目 | 期間 | コスト |
|---------|:----:|:----:|
| VP策定・QA体制整備 | 3ヶ月 | 0.15億円 |
| CSV実施（IQ/OQ/PQ） | 6ヶ月 | 0.40億円 |

| SOP・運用文書整備 | 3ヶ月 | 0.10億円 |

## 4-8. AIセキュリティ技術実装（10領域フレームワーク：Phase 4担当4領域）

### 領域2：AIサプライチェーンセキュリティ

| 対策項目 | 日本化薬 医薬品AIへの適用 | 実装優先度 |
|---------|----------------------|:--------:|
| **外部LLMモデルの検証** | Claude API / Azure OpenAI利用前に、Anthropic・Microsoftのセキュリティ認証（SOC2 Type II、ISO 27001）を確認。年次更新を義務化 | ★★★★★ |
| **Schrödinger等外部AIライブラリの脆弱性スキャン** | ADCリンカーAIで使用するSchrödinger FEP+・OSS分子生成ライブラリに定期的な依存関係スキャン（Snyk/Dependabot等）を適用 | ★★★★☆ |
| **AIモデル改ざん検知（モデルハッシュ検証）** | 本番AIモデルのデプロイ時に、学習済みモデルファイルのSHA-256ハッシュ検証を必須化。不一致時は自動ロールバック | ★★★★★ |
| **サードパーティデータソースの信頼性評価** | 外部文献DB（PubMed, ClinicalTrials.gov等）からのデータ取込み時に、ソース認証とデータ品質チェックを自動実施 | ★★★☆☆ |

### 領域5：プライバシー保護技術（PETs）

| 技術 | 適用場面 | 日本化薬での実装方針 |
|-----|---------|------------------|
| **差分プライバシー（DP）** | 患者臨床データを用いたAIモデル学習 | ε=1.0〜5.0の差分プライバシーを適用し、個別患者情報が推論されないことを数学的に保証 |
| **連合学習（Federated Learning）** | 複数医療機関との共同AI研究 | 患者データを医療機関外に出さずにモデル学習を可能にする。アカデミア連携時の標準プロトコルとして採用 |
| **秘密計算（Homomorphic Encryption）** | ADCリンカー化合物データの外部パートナー共有 | 化合物構造を暗号化したまま外部サービスで計算可能に。Schrödingerとの連携で部分的に検討 |
| **k-匿名化・l-多様性** | MR・患者属性データのAI分析 | CRMデータをAI分析する際、医師・患者の個人特定を防ぐ最小匿名化基準を定義（k≥5） |

### 領域7：データリネージ

| 管理対象 | リネージ追跡の要件 | 実装ツール |
|---------|----------------|:--------:|

| **モデルバージョンとデータの紐付け** | どのモデルバージョンが、いつ、どのデータで学習されたかをMLflowで完全追跡 | MLflow（GxP: Domino） |

### 領域9：Zero Trust for AI

| Zero Trust原則 | AIシステムへの適用 | 日本化薬実装方針 |
|-------------|----------------|--------------|
| **最小権限の原則** | AIモデルAPIへのアクセスは必要なユースケースごとに個別APIキーを発行。MR向けAIとADC設計AIのキーを完全分離 | Azure RBAC + API Management |
| **継続的認証（Never Trust, Always Verify）** | AIシステムへのアクセスはセッション中も定期的な再認証を要求。長時間セッションでの不正利用を防止 | Azure Conditional Access |
| **マイクロセグメンテーション** | Layer A/B/Cの3層間でのデータ・API通信を厳格に制御。Layer CのADC設計AIへはLayer Aからの直接アクセスを禁止 | Azure VNet + NSG |
| **AIリクエスト/レスポンスの検査** | 全AIシステムへの入出力をリアルタイムでスキャン。機密化合物データの意図しない送信を自動検知・ブロック | DLP + Azure AI Content Safety |
