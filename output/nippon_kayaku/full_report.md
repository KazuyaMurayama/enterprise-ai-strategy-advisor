## 最優先で着手すべきアクション TOP3

### 1. [推奨] 医薬品AI CoE設立（経営直轄・専任5名）
- **理由**: 日本化薬の医薬品事業規模では大型専任組織は不要だが、「司令塔」なきAI投資は各部門の散発PoC乱立に終わる。5名のコア専任チーム（医薬品ドメイン×AI）を経営直轄で設置し、全AI取組みの優先順位・予算・品質を一元管理する
- **担当**: 医薬品事業本部長直轄
- **期限**: 2026年Q2（3ヶ月以内）にCoE発足、専任メンバー選定完了

### 2. [推奨] MR支援AIアシスタント展開（Quick Win #1）
- **理由**: 400名のMRが抱える訪問準備・資料作成工数（推定週10-12時間/人）をAIで50-70%削減。GxP対象外で規制ハードル最低。3ヶ月でPoC完了・6ヶ月で全MR展開が可能な最速ROI創出策。MRの訪問集中度向上で医師カバレッジが拡大し、売上向上にも直結する
- **担当**: 営業部門 + IT部門
- **期限**: 2026年Q3（6ヶ月以内）に全MR展開完了

### 3. [推奨] ADCリンカー・ペイロード最適化AI着手（戦略的最重要）
- **理由**: 日本化薬の最大の競争優位であるADC技術に生成AI（分子設計AI）を適用し、リンカーの安定性・ペイロードのTI（治療指数）を同時最適化する。先行企業（AstraZeneca, Daiichi Sankyo等）との技術格差が2028年以降に不可逆的に拡大する前に着手が必須。外部AI創薬パートナー（Schrödinger等）との提携から開始し、リスクを限定しながら進める
- **担当**: 研究本部 + CoE
- **期限**: 2026年Q4（9ヶ月以内）にパイロット研究開始

## 8-2. 最重要リスクの詳細と対策

---

### R01：ADCリンカーデータの機密漏洩

| 項目 | 内容 |
|-----|-----|
| **リスク詳細** | 外部AIプラットフォーム（Schrödinger等）APIへのADC化合物データ送信、クラウドストレージへの保存時の暗号化不備、パートナー企業の情報管理不備等により、日本化薬の最重要IP（リンカー設計ノウハウ・未公開化合物データ）が競合に漏洩する |
| **インパクト** | コアADC競争優位の永久喪失。特許出願前データ漏洩の場合は特許権取得不能。ライセンス収入の長期的毀損（累計影響額: 数百億円） |
| **対策①** | ADCリンカー設計AIは外部APIを使用せず、オンプレミスGPUサーバーで完結実行。Schrödingerはローカル実行モード契約のみ |
| **対策②** | データ分類ポリシー策定（機密ランク別: 社外秘S/A/B）。ランクS（ADC未公開化合物）はオフライン環境のみでのAI処理を義務化 |
| **対策③** | 外部パートナー契約にデータ使用制限・秘密保持条項（厳格なNDA）を必須化。違反時の損害賠償条項を明記 |
| **対策④** | DLP（Data Loss Prevention）ツール導入。ADC関連データの外部転送を自動検知・遮断 |
| **残留リスク** | 内部犯（悪意ある社員）によるデータ持ち出し。完全排除は困難だが、アクセスログの監査と権限の最小化で抑制 |

---

### R02：PMDA審査での追加要求・遅延

| 項目 | 内容 |
|-----|-----|
| **リスク詳細** | AI活用した臨床試験データ・CTD文書に対し、PMDAが追加の説明資料・再試験を要求。AI活用の透明性・説明可能性が不十分な場合、審査期間が6-12ヶ月延長するリスク |
| **インパクト** | 申請1件当たりの追加費用: 3-8億円。上市遅延1ヶ月当たり機会損失: 5-15億円（製品規模による） |
| **対策①** | Phase 0から薬事部門がPMDA事前相談を実施。「AI活用計画書」を事前提出し、審査方針を早期に確認 |
| **対策②** | AI活用箇所の完全な記録（説明可能AI：SHAP値・Attention可視化等）を申請文書に添付する標準手順を確立 |
| **対策③** | 業界団体（JPMA等）のAI活用WGに参加し、PMDA対応の業界ベストプラクティスを先取り |
| **モニタリング指標** | PMDA事前相談の実施件数、AI活用申請でのPMDA照会事項件数（目標: 非AI申請比で増加なし） |

## 8-4. AI倫理ガバナンス

### 日本化薬 医薬品AI倫理の3つの柱

#### 柱1：患者データの適正使用原則

| 原則 | 内容 |
|-----|-----|
| **目的外使用の禁止** | 臨床試験データをAI学習に使用する場合は、試験プロトコルで明示的に定められた目的の範囲内のみ |
| **匿名化の徹底** | クラウドAI処理前に個人識別子を完全除去。Re-identification攻撃への対策（K-匿名化・差分プライバシー） |
| **インフォームドコンセント** | AI活用目的の研究・分析に患者データを使用する場合、インフォームドコンセントに明示的に記載 |
| **データ最小化** | AI学習に使用するデータは目的達成に必要最小限に留める（GDPR・個人情報保護法準拠） |

#### 柱2：ADC毒性AI判断への過度な依存防止

| 原則 | 内容 |
|-----|-----|
| **人間のオーバーサイト義務** | AIが「安全」と予測した化合物であっても、必ず専門家（毒性学者）の独立した評価を最終判断として要求 |
| **不確実性の明示** | AI毒性予測に必ず信頼区間・予測精度を表示。研究者がAI限界を認識した上で判断できる環境を確保 |
| **説明可能AI（XAI）の採用** | 毒性予測に影響した分子構造特徴を可視化。ブラックボックス判断を排除し、科学的根拠の確認を可能にする |

#### 柱3：MR評価へのAI活用の透明性

| 原則 | 内容 |
|-----|-----|
| **評価への利用の明示** | AI活用データが人事評価に影響する場合は、事前に評価基準・指標を全MRに開示 |
| **異議申し立て権** | AI算出スコアに異議があるMRは、上司＋HR部門への説明要求権を有する |
| **ネガティブ評価への単独使用禁止** | AI指標のみを根拠とした不利益な人事決定は禁止。必ず複数情報源・人間判断と組み合わせる |

## 8-7. AIセキュリティ高度実装（10領域フレームワーク：Phase 8担当6領域）

### 領域1：LLMセキュリティ

| 攻撃ベクター | リスク | 日本化薬での対策 |
|-----------|------|--------------|
| **プロンプトインジェクション** | MR AIアシスタントへの悪意ある入力で機密情報を抽出 | システムプロンプトの強化（インストラクションを上書きできない構造）。入力値のサニタイズ（禁止パターン検出） |
| **間接インジェクション（RAG経由）** | RAGが参照する外部文書に埋め込まれた攻撃プロンプト | RAG参照文書の取込み時スキャン。Azure AI Content Safetyによるリアルタイム検査 |
| **モデル逆転攻撃（Training Data Extraction）** | LLMへの反復クエリで学習データ（患者情報等）を抽出 | クエリレート制限・異常クエリパターン検知。患者データの学習への使用を原則禁止 |
| **モデル盗用（Model Stealing）** | APIへの大量クエリで社内ファインチューニングモデルを複製 | APIアクセスの認証強化。疑わしいクエリパターン（体系的な探索）の自動検知・ブロック |

### 領域4：AI特有インシデントレスポンス

#### AI障害・誤判定インシデント対応手順

```
【レベル1: AI誤出力（軽微）】
  → 当該AIシステムのフラグ立て → CoEへ報告 → 24時間以内にパッチ対応
  例: MR AIアシスタントが古い臨床データを参照

【レベル2: AI誤判定（業務影響あり）】
  → 当該AIシステムの一時停止 → CoE + 業務部門長へ即時報告
  → 影響範囲の特定（何件の判断が影響を受けたか） → 修正後に段階復旧
  例: 製造品質AIが正常バッチを不良判定し廃棄

【レベル3: AI重大インシデント（GxP対象・患者影響懸念）】
  → AIシステム全停止 → CoE + QA + 法務 + 経営層へ即時報告
  → PMDAへの報告要否を12時間以内に判断（薬事部門が判断）
  → 外部調査委員会の設置 → 再発防止策の策定・実施
  例: CTD生成AIが誤った臨床データでドラフトを作成し申請に混入
```

#### AI障害時の手動代替手順（フォールバック）

| AIシステム | 障害時の代替手順 | 代替手順の訓練頻度 |
|---------|--------------|:--------------:|
| MR AIアシスタント | 既存CRMシステムの通常利用に切り替え | 半期1回 |
| CTD自動生成AI | 薬事専門家による手作業作成（SOP完備） | 申請前に必ず確認 |
| 製造品質AI | 手動サンプリング検査プロトコルへ即時切替え | 月1回（製造部門OJT） |
| 患者選択AI | 試験医師による従来の適格性判断に切替え | 試験開始前に確認 |

### 領域6：説明可能性（XAI）

| AI適用場面 | XAI手法 | 提供する説明の形式 |
|----------|:-------:|----------------|
