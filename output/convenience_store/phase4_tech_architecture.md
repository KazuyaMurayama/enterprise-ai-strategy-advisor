# フェーズ4: 技術アーキテクチャ戦略 — レガシーPOS資産を活かすハイブリッドAI基盤

作成日: 2026-05-20
最終更新日: 2026-05-20

## 1. このセクションの要点

- **Build/Buy/Partner判断**: 需要予測コアはBuild（競争源泉）、LLM基盤はBuy（OpenAI/Azure/Anthropic）、店舗AIカメラはPartner（NEC/富士通/AWL）
- **ハイブリッド移行が核心**: 1990年代からのレガシーPOS・トータルシステム資産を活かしながら、クラウドネイティブAI基盤（Snowflake/Databricks等）へ段階的に移行することが技術戦略の根幹
- **投資構造**: 技術投資108億円のうちデータ基盤刷新35億円・AIカメラ展開30億円が最大項目。MLOps・エッジAI・セキュリティを含む包括的投資で競争優位を構築

## 2. 技術アーキテクチャ全体像

### 5層アーキテクチャ構成

| レイヤー | 主要構成要素 | 技術選択肢（推奨） | 投資規模（3年） |
|:---:|---|---|---:|
| **L1 データ収集層**（Edge/Store） | POSレジ、IoT Gateway、AIカメラ、電子棚札（ESL）、エッジ推論チップ | Hailo-8 / Jetson Orin / AWL Edge | 30億円（AIカメラ） |
| **L2 データ統合・レイクハウス層** | Data Lakehouse、ストリーミング基盤、ETL/ELT、データカタログ | Snowflake or Databricks / Kafka / dbt+Airflow / Apache Atlas | 35億円 |
| **L3 特徴量・ML基盤層** | Feature Store、ML Platform、MLOps | Feast or Tecton / Vertex AI or SageMaker / MLflow | 10億円 |
| **L4 AI/LLM基盤層** | 生成AIゲートウェイ、マルチモデル管理、RAG基盤、Fine-tune環境 | LiteLLM or Kong AI Gateway / pgvector or Pinecone / Stockmark LLM | 15億円（API料） |
| **L5 アプリケーション層** | FCオーナーAIコパイロット、本部MD・SVダッシュボード、公式アプリAPI、ESL連携 | React Native / Next.js / FastAPI / LINE連携 | 11億円（エッジAI含む） |

### 2-1. データ収集層（Edge/Store Layer）詳細

店舗15,000店の末端から1日約1億トランザクションを収集する基盤。既存のPOSレジ（富士通/NEC/東芝TEC系）はIoT Gatewayを中継点として活用し、フルリプレースコストを回避する。

- **POSレジ連携**: 既存レジのログをIoT Gateway（店舗内ラズパイ相当の軽量サーバ）経由でクラウド送信。通信帯域が細い店舗（ADSL残存）はLTE/5Gモバイルルータへの移行をYear1中に完了
- **AIカメラ**: AWL/NEC/i-PROのエッジ推論対応カメラを全店展開（15,000店×2台＝30,000台）。エッジチップ（Hailo-8またはJetson Orin）により店舗内リアルタイム処理（棚在庫検知・入店カウント・防犯）
- **電子棚札（ESL）**: SES-imagotag/パナソニック製ESLと連携。ダイナミックプライシング実行時の価格変更をクラウドからESLへ秒単位で反映
- **エッジ推論の方針**: 映像・センサーデータの生データをクラウドへ全量送信するとコスト・帯域が逼迫するため、エッジ側で集計・特徴量抽出を実施し、差分データのみをクラウドへ転送

### 2-2. データ統合・レイクハウス層詳細

**Snowflake vs. Databricks コスト比較**

| 評価軸 | Snowflake | Databricks |
|---|---|---|
| **課金モデル** | クレジット従量制（ストレージ+コンピュート分離） | DBU（Databricks Unit）従量制 |
| **日本語サポート** | 充実（国内SE常駐） | 充実（AWS/Azure経由） |
| **機械学習統合** | 外部ツール依存（Snowpark ML） | MLflow/Delta Lake完全統合 |
| **推定3年コスト**（本規模） | 約12-15億円 | 約10-13億円 |
| **移行難易度（Oracle/Teradata から）** | 中（SQL互換性高） | 中-高（Spark知識要） |
| **推奨判断** | 分析BI中心ならSnowflake | ML/AIパイプライン中心ならDatabricks |

[推奨] ML活用の比重が高い本件ではDatabricksを第一候補とする。ただしBI部門の既存Snowflake資産がある場合はデュアル構成（Snowflake for BI + Databricks for ML）も許容。

- **リアルタイムストリーミング**: Kafka（セルフホスト on Kubernetes）またはAmazon Kinesis。1億トランザクション/日の処理にはKafkaパーティション数200以上を推奨
- **ETL/ELT**: dbt（変換）+ Apache Airflow（オーケストレーション）の標準スタック。POS独自フォーマット→標準スキーマへの変換ロジックをdbt modelとして管理
- **データカタログ**: Apache Atlas（OSSで総保有コスト最適）またはCollibra（エンタープライズSaaS）。全テーブルの定義・血統（Lineage）・オーナーをメタデータ登録必須

### 2-3. 特徴量・ML基盤層詳細

- **Feature Store（Feast or Tecton）**: 需要予測・推薦・異常検知の特徴量を一元管理。オンライン（低レイテンシ推論用）・オフライン（学習用）の二重ストアを構築
- **ML Platform**: Vertex AI（GCP）またはSageMaker（AWS）。マネージドサービスを採用しMLOpsインフラ構築工数を削減。ハイパーパラメータ最適化・分散学習・モデルデプロイを統一管理
- **MLOps（MLflow/Weights&Biases）**: 実験管理・モデルバージョン管理・A/Bテスト結果記録。MLflowはOSSで内製可。W&Bはリッチな可視化が強み（年間SaaS費用：約2,000万円/100ユーザ）

### 2-4. AI/LLM基盤層詳細

- **生成AIゲートウェイ**: LiteLLMまたはKong AI Gatewayを全社統一エンドポイントとして配備。コスト管理・レート制限・フォールバック（OpenAI障害時にAnthropicへ自動切替）を一元制御
- **マルチモデル戦略**: 用途別に最適モデルを選定（詳細は第4節参照）
- **RAG基盤**: ベクトルDBにpgvector（PostgreSQL拡張・既存DB資産活用）またはPinecone/Qdrant（マネージド）。FC向けQAシステム・本部規程検索・商品情報検索に適用
- **Fine-tuning環境**: 日本語食品ドメイン特化用途にStockmark社国産LLMのFine-tuneを検討。汎用用途はOpenAI Fine-tuning APIで対応

### 2-5. アプリケーション層詳細

- **FCオーナー向けAIコパイロット**: タブレット/スマートフォンアプリ（React Native）。発注提案・クレーム対応・労務管理を統合。LINE Business API経由の通知も実装
- **本部MD・SV向けダッシュボード**: Next.js + Tableau/Looker。商品MD担当者向け需要予測可視化・PB商品開発支援ツール・SV向け異常店舗検知アラート
- **公式アプリAPI連携**: ローソンアプリ/ファミペイ相当の顧客IDと連携したパーソナライズ推薦API（FastAPI）
- **ESL連携API**: ダイナミックプライシングエンジンから電子棚札管理サーバへのリアルタイム価格更新

## 4. 基盤モデル選定マトリクス

| タスク | 推奨モデル | 日本語性能 | コスト感 | 選定根拠 |
|---|---|:---:|:---:|---|
| **FCオーナーコパイロット（対話・業務支援）** | Claude 4 Sonnet | ◎ | 中 | 日本語優秀・安全性高・長文コンテキスト対応・API安定性 |
| **商品コンセプト・PB企画生成** | GPT-4o | ○ | 中 | マルチモーダル（パッケージ画像→コンセプト）・創造性 |
| **PB商品説明文・チラシコピー** | Stockmark国産LLM or GPT-4o Fine-tune | ◎ | 低-中 | 日本語食品ドメイン特化。Stockmark社「Tanuki-8B」等を検討 |
| **需要予測本体** | カスタムTransformer / LightGBM | — | 低 | LLMは補助的用途（季節コメント生成等）。予測本体は専用MLモデル |
| **画像認識（棚割チェック・防犯）** | Claude Vision or GPT-4V + 専用CNN | ○ | 中 | 汎用VLMを前段フィルタに、棚割特化CNNを後段に組み合わせ |
| **多言語接客UI（訪日外国人対応）** | DeepL API + LLM後処理 | ◎ | 低 | DeepLの高精度翻訳にLLMで文脈補正を追加 |
| **本部内部ナレッジQA（RAG）** | Claude 3.5 Haiku（低コスト） | ◎ | 低 | 規程・マニュアル検索は低コストモデルで十分。RAGで精度補完 |
| **異常検知アラート文生成** | GPT-4o-mini or Claude 3.5 Haiku | ○ | 低 | 定型文生成なので軽量モデルで対応。月間コスト抑制優先 |

[推奨] 本番用途は単一モデルベンダーへの依存を排除するマルチモデル戦略を採用。LLMゲートウェイ（LiteLLM等）でコスト・性能・可用性を動的に最適化。OpenAI障害時のフォールバック先としてAnthropicを常時待機。

| **FC加盟店データ収集同意** | FC契約書改訂。データ収集・AI学習への利用に関する明示的同意取得 | 法務・FC推進部門 | Year1 Q1 |

### 5-2. ID-POSデータ活用方針

年間300億トランザクションのID-POSデータは最大の競争資産である一方、プライバシーリスクも内包する。

- **準識別子リスク評価**: 購買履歴×時刻×店舗の組み合わせにより個人特定が可能となるリスクを定量評価（k-匿名性モデルで閾値k≥10を設定）
- **匿名化・仮名化処理**: k-匿名性・l-多様性に準拠した処理パイプラインをData Lakehouse入口に実装。個人IDはハッシュ化（HMAC-SHA256）し、元データへの逆引き不可の環境を整備
- **目的別データアクセス制御**: 需要予測用・マーケティング用・商品開発用の各データマートを分離管理。第三者提供は原則禁止（同意取得済みの提携先のみ例外）

### 5-3. 特徴量エンジニアリング

| ユースケース | 主要特徴量カテゴリ | 特徴量例 |
|---|---|---|
| **需要予測** | 時系列・外部データ・立地 | SKU別売上時系列（過去52週）、気象（気温・降水量）、人口属性、周辺施設（学校・オフィス）、曜日・祝日・イベント |
| **パーソナライズ推薦** | 顧客行動・嗜好 | 購買頻度・カテゴリ嗜好・時間帯・支払い方法・過去推薦クリック率 |
| **異常検知** | センサー・物理データ | 電力消費異常（冷蔵ショーケース故障検知）、在庫差異（POSvs物理棚卸し）、カメラ動体パターン |
| **棚割最適化** | 空間・売上データ | 棚位置別売上、陳列面積、SKU共起購買、季節性指数 |

### 5-4. データリネージ（セキュリティ10領域 #7：必須）

ID-POSデータの完全追跡パイプライン（取得→推論→フィードバックの全過程）:

| ステップ | システム | メタデータ管理内容 | 不可逆記録方法 |
|:---:|---|---|---|
| **1. 収集** | POSレジ（富士通/NEC） | 収集日時・店舗ID・POS端末ID・生ログハッシュ | ブロックチェーン台帳への書込み（Hyperledger Fabric） |
| **2. 前処理** | IoT Gateway / Kafka | 変換ルールバージョン・処理日時・レコード件数 | Kafkaトピックのイミュータブルログ（保持期間7年） |
| **3. 保存** | Data Lake (Databricks) | スキーマバージョン・パーティション・ACID保証 | Delta Lakeのトランザクションログ（変更不可） |
| **4. 特徴量化** | Feature Store (Feast) | 特徴量定義バージョン・生成日時・ソーステーブル | Feastのfeature store registryに変更履歴を自動記録 |
| **5. 学習** | ML Platform (Vertex AI) | 実験ID・使用特徴量・ハイパーパラメータ・学習データHash | MLflow Experiment Tracking（全パラメータを不変ログに保存） |
| **6. 登録** | Model Registry (MLflow) | モデルバージョン・性能指標・承認者・デプロイ日時 | Git-taggingとModel Registryのイミュータブル履歴 |
| **7. 推論** | Inference API (FastAPI) | リクエストID・入力ハッシュ・出力・レイテンシ | 推論ログをS3/GCS（Write Once Read Many設定）に保存 |
| **8. 出力** | アプリケーション | 表示先（FC/本部）・表示内容・ユーザアクション | アプリケーションログ（Datadog/CloudWatch） |
| **9. フィードバック** | Feedback Pipeline | 予測精度・実績値・ドリフト指標 | 定期バッチでModel Registryにフィードバックメトリクス記録 |

各ステップのメタデータはApache Atlasに集約し、データカタログ上でE2Eの血統を可視化。監査要請時には任意のデータポイントの来歴を2営業日以内に提示できる体制を構築する。

## 6. セキュリティアーキテクチャ（CLAUDE.md指定 Phase 4担当4領域）

### 6-1. AIサプライチェーンセキュリティ（セキュリティ10領域 #2）

[推奨] 採用する全AIモデル・ライブラリのサプライチェーンリスクを調達段階から管理する。

**Model Card 必須確認項目**

| 確認項目 | OpenAI（GPT-4o） | Anthropic（Claude 4） | Google（Gemini） | Stockmark |
|---|:---:|:---:|:---:|:---:|
| 学習データソース開示 | 部分開示 | 部分開示 | 部分開示 | 国内コーパス明示 |
| 安全性評価結果 | 公開 | 公開 | 公開 | 限定公開 |
| 利用規約（商用可） | ○ | ○ | ○ | ○ |
| データ学習オプトアウト | API利用時○ | API利用時○ | API利用時○ | — |

**AI-SBOM（AI部品表）管理体制**

- 使用中の全AIモデル・バージョン・学習データソース・ライセンス・依存OSSライブラリを一元管理台帳に登録
- 台帳更新は月次実施。新規モデル採用時は法務・セキュリティ部門の承認を必須化
- OSSモデル（Hugging Face等）の脆弱性スキャン: Dependabotによる依存関係監視、ClamAVによるモデルファイルのマルウェアスキャンを月次実施
- ベンダーのAIセキュリティ認証確認: ISO/IEC 42001（AI管理システム）、SOC2 Type IIの取得状況を調達評価基準に追加

### 6-2. プライバシー保護技術 PETs（セキュリティ10領域 #5）

**差分プライバシー（Differential Privacy）の適用検討**

| 適用対象 | プライバシーバジェット(ε) | 精度影響試算 | 適用判断 |
|---|:---:|---|:---:|
| ID-POS集計レポート（対外開示用） | ε=1.0 | 需要予測精度低下±2%以内（試算） | **Year2から適用** |
| 内部分析用マート（第三者提供なし） | ε=5.0〜10.0 | 精度影響軽微（±0.5%以内） | **Year1から適用** |
| モデル学習（DP-SGD） | ε=3.0〜8.0 | 予測精度低下5-10%（許容範囲評価必要） | **Year2以降に評価** |

[推奨] まずε=5.0〜10.0の内部マートから適用開始。精度影響を実測後、ε=1.0の対外開示用DPへ段階的に移行する。

**連合学習（Federated Learning）の検討**

- 方式: 店舗端末でローカルモデル学習→勾配のみサーバに送信→グローバルモデル更新（FedAvg）
- メリット: 生データをサーバに集約せず、FC加盟店との信頼醸成に貢献
- 実装コスト試算: 追加5〜8億円（15,000店舗への展開インフラ、通信コスト含む）
- 前提条件: 店舗端末の計算資源確保（GPU非搭載端末では現実的でない）。Year3以降のパイロット検討を推奨

**秘密計算（Homomorphic Encryption）の評価**

- 現時点では演算コストが通常比100〜1,000倍であり、1億トランザクション/日の本番処理への適用は困難
- 2030年以降の技術成熟を待って再評価。小規模・低頻度の集計タスクでのPoC実施を2028年頃に計画

### 6-3. データリネージ（セキュリティ10領域 #7）

詳細は前節「5-4. データリネージ」を参照。セキュリティ視点での追加要件:

- 全データ変換ステップへの改ざん検知（ハッシュチェーン）実装
- 不正アクセス・データ操作の監査ログをSIEMと連携し、リアルタイム異常検知を実施
- 個人情報を含む処理ステップは別ログに分離管理（個人情報保護法対応）

### 6-4. Zero Trust for AI（セキュリティ10領域 #9）

[推奨] AIサービスへのアクセスは「信頼しない、常に検証する」ゼロトラスト原則を徹底適用する。

**レイヤー別Zero Trust実装**

| レイヤー | 認証・認可方式 | 実装内容 |
|---|---|---|
| **店舗端末（POSレジ）** | Device Identity証明書 + 端末登録 | 各POSレジに固有のクライアント証明書を発行。未登録端末からの接続は即座に拒否 |
| **API層（LLMゲートウェイ）** | OAuth2.0 / JWT + スコープ限定認可 | 全API呼び出しに Bearer Token必須。スコープは最小限（例: demand-forecast:read のみ） |
| **ネットワーク分離** | マイクロセグメンテーション | 需要予測API・LLMゲートウェイ・個人情報DB・管理系を別VLANに分離。相互通信はFirewall Policy で明示許可のみ |
| **外部LLMベンダー接続** | VPN + MFA + API Key Rotation | VPNトンネル経由でOpenAI/Anthropic APIへ接続。APIキーは月次ローテーション自動化（HashiCorp Vault使用） |
| **FCオーナーアプリ** | アプリ側アクセス制御 | FCオーナーは自店舗データのみ閲覧可能。本部システム・他店舗データへのアクセスをアプリ層・API層の二重で遮断 |
| **本部社員アクセス** | Identity Provider（Azure AD/Okta）+ 条件付きアクセス | 職位・部門に応じたRBACポリシー。機密データへのアクセスはMFA必須+デバイスコンプライアンスチェック |

## 8. MVPアーキテクチャ（Phase 1 Quick Winのための最小構成）

Phase 0〜1で最速のROI実証を行うための最小技術スタック。既存資産を最大限活用し、新規インフラへの依存を最小化する。

| MVP | 技術スタック | 期間 | 期待成果 |
|---|---|:---:|---|
| **需要予測MVP** | 既存DWH（Oracle/Teradata）+ Python LightGBM + シンプルなダッシュボードAPI（FastAPI+Streamlit） | 6ヶ月 | 廃棄率5%削減のPOC。既存DWHから移行せずに成果を示す |
| **エネルギー最適化MVP** | 既存IoTセンサー（冷蔵ショーケース） + Rulebook（閾値アラート） + クラウドアラート（CloudWatch/Datadog） | 3ヶ月 | 電力コスト3%削減の即効性。ハードウェア追加投資ゼロで実現 |
| **FCコパイロットMVP** | Claude 4 Sonnet API + RAG（本部FAQ/規程PDF） + LINE Business API or タブレットWebアプリ | 4ヶ月 | FC問い合わせ対応時間50%削減のPOC。LLMゲートウェイ不要で試作可能 |

**MVPアーキテクチャの原則**

1. **既存資産のフル活用**: 既存DWH・IoTセンサー・POSデータを活用し、Data Lakehouseへの移行前に成果を出す
2. **最短Time-to-Value**: 3〜6ヶ月でROIを実証し、本格投資の経営承認を取得する
3. **段階的移行**: MVP成果を基にData Lakehouseへの段階移行を正当化。ビッグバン移行リスクを回避

## 9. So What? — 技術戦略の核心メッセージ

> **「レガシーPOS資産はゴミではなく宝。30年分のID-POSデータを新AI基盤で『活性化』することが技術戦略の核心」**

セブン-イレブン・ジャパンが需要予測を完全内製化し業界最高の廃棄率・欠品率を実現しているように、コンビニのAI競争優位はデータ量と予測アルゴリズムの内製能力に依存する。本技術アーキテクチャは以下の3点を技術戦略の根幹に置く:

1. **ハイブリッド移行**: レガシーPOSを即廃棄せず、IoT Gatewayで現行システムとクラウドAI基盤を橋渡し。移行リスクを最小化しながら段階的にモダナイズ
2. **マルチモデルLLM戦略**: 単一ベンダー依存を排除。LLMゲートウェイで複数モデルを動的切替し、可用性・コスト・性能を三角最適化
3. **セキュリティ・バイ・デザイン**: ゼロトラスト・データリネージ・プライバシー保護技術（PETs）をアーキテクチャ設計段階から組み込み、後付けコスト（リファクタリング費用は初期組込みの3〜5倍）を回避

この3原則を貫くことで、30年分の資産を競争優位に変換し、MBB水準の「技術で勝つ」AI戦略を実現する。
