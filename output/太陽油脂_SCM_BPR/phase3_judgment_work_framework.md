## 3-1. フレームワーク設計思想：なぜ「定義」が先か

### 本プロジェクトが犯してはならない思考順序

```
【危険な思考順序（やってはいけない）】
  削減目標（▲7名）→ 削減できる業務を探す → 判断業務を後付けで守る

【正しい思考順序（本フレームワーク）】
  判断業務を客観的に定義・保護 → 残余業務の自動化可否を判定 → 削減工数が積み上がる → 人員数が導き出される
```

この順序を守ることで、「AIで全員削減するのでは」という現場の誤解を解消し、役員会での論理的な承認を可能にする。

### 3軸評価モデルの設計原則

| 設計原則 | 内容 |
|---|---|
| **客観性** | 感覚的な「重要業務」を排除し、3軸の点数で分類する |
| **監査可能性** | 将来の労務問題に備え、分類根拠を文書化・保存する |
| **変動対応** | 技術進化により分類が変わりうることを前提に、再評価プロセスを組み込む |
| **現場合意** | 評価は管理職だけでなく担当者ヒアリングを経て確定する |

## 3-2. 3軸スコアリングモデル

### 軸の定義と採点基準

#### 軸A: 情報完全性（Completeness of Information）
> 「この業務の判断に必要な情報は、構造化データとして100%揃っているか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | 必要情報がすべて構造化データ（DB/CSV）で存在 | 標準品の在庫残高確認、定型受注の価格参照 |
| **1点** | 主要情報は構造化、一部は文書/口頭情報を参照 | 通常の生産計画（BOMあるが一部現場確認要） |
| **2点** | 構造化データ5割＋非構造化情報（メール・商談内容）5割 | MC品目の需要予測（市場感・顧客ヒアリング込み） |
| **3点** | 市場感・関係性・暗黙知が判断の主要情報源 | 不採算品の製造継続判断、新原料調達先選定 |

#### 軸B: 判断再現性（Reproducibility of Decision）
> 「同じ情報を与えれば、担当者が誰でも同じ判断をするか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | ルール・閾値が完全定義されており誰でも同一結果 | 安全在庫下回りアラート発報、定型受注入力 |
| **1点** | 大半のケースはルール適用、例外時は判断が分かれうる | 標準品MRP実行、通常配車割当 |
| **2点** | ルールはあるが経験・文脈で判断が変わる（±20%程度） | 需給逼迫時の優先順位付け、クレーム対応方針 |
| **3点** | 担当者の経験・判断が本質、標準化不可 | 共同製品の市場価値判断、新規顧客の与信判断 |

#### 軸C: 例外発生頻度（Frequency of Exceptions）
> 「想定外の状況・判断への割り込みはどのくらい発生するか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | 例外はほぼゼロ（月1件未満） | 週次定型レポート出力、FAX受注の定型入力 |
| **1点** | 例外は月数件、ルールで対処可能 | 原料価格変動時の発注量調整、返品処理 |
| **2点** | 週次で例外が発生し、都度判断が必要 | バッチ製造の歩留まり異常対応、配送遅延調整 |
| **3点** | 日常的に想定外が発生し、即時判断が事業継続に直結 | 原料タイト時の製造優先品目決定、大口解約対応 |

### 3軸合計スコアによる4分類

```
スコア計算式: 合計点 = 軸A + 軸B + 軸C （最大9点）

  0〜2点 ┌─────────────────────────────────────────────┐
         │  【実行業務】                                  │
         │  AI/RPA完全自動化対象。人間の介入なしで処理可能  │
         │  → Phase 4でRPA/OCR化計画を策定               │
         └─────────────────────────────────────────────┘

  3〜4点 ┌─────────────────────────────────────────────┐
         │  【準判断業務】                                │
         │  AI提案＋人間最終承認（Human-in-Loop）         │
         │  → Phase 4で生成AI+承認フロー設計              │
         └─────────────────────────────────────────────┘

  5〜6点 ┌─────────────────────────────────────────────┐
         │  【判断業務】                                  │
         │  人間主体、AI情報提示支援のみ許容              │
         │  → 5/20役員会の承認対象（保護業務として確定）   │
         └─────────────────────────────────────────────┘

  7〜9点 ┌─────────────────────────────────────────────┐
         │  【高度判断業務】                              │
         │  人間必須。AI代替不可・外注不可               │
         │  → 5/20役員会の承認対象（絶対保護）            │
         └─────────────────────────────────────────────┘
```

## 3-3. 5つの客観テスト

スコアリングの妥当性を検証するための追加テスト。スコアが3〜4点（境界値）の業務に適用する。

| テスト | 問い | 判断業務と判定する条件 |
|:---:|---|---|
| **①過去ログ再現テスト** | 過去の判断ログから同一の結論が機械的に再現できるか？ | 再現できない場合 → 判断業務 |
| **②説明可能性テスト** | 判断根拠を第三者に論理的に説明できるか（Yes/No以外）？ | 説明に「感覚・経験」が必要な場合 → 判断業務 |
| **③例外記録テスト** | 直近12ヶ月の例外ケースが10件以上あるか？ | 10件以上の場合 → 準判断業務以上 |
| **④誤り事後検証テスト** | AI判断が誤った場合、事業損害が発生するまでの検出猶予は24h以上あるか？ | 24h未満の場合 → 判断業務（人間監視必須） |
| **⑤代替可能性テスト** | その担当者が突然不在になった場合、別の担当者が同等の判断を即座にできるか？ | できない場合 → 高度判断業務 |

## 3-4. AI/RPA化判定ルール（4条）

### Rule 1: 強制人間判断条件（いずれか該当でAI代替禁止）
- [ ] 判断が法令・コンプライアンス上の責任を伴う
- [ ] 判断が従業員の雇用・評価に直接影響する
- [ ] 誤判断の損害額が単月売上の1%超になる可能性がある
- [ ] 顧客・サプライヤーとの関係維持に「人の判断」が期待されている

### Rule 2: AI支援許容条件（準判断業務の要件）
- [ ] AI出力を人間が最終確認・承認するフローが設計されている
- [ ] AI判断の根拠（説明性）が担当者に提示される
- [ ] 誤判断時のロールバック手順が定義されている
- [ ] 月次でAI精度をレビューする担当者が指名されている

### Rule 3: 完全自動化条件（実行業務の要件）
- [ ] 処理対象データがすべてデジタル・構造化されている
- [ ] 判断ロジックがIf-Then-Else形式で完全記述できる
- [ ] 例外発生時に人間へのエスカレーションルートが自動発動する
- [ ] 月次の自動化精度が98%以上を維持できることが確認されている

### Rule 4: 段階的移行ルール
```
実行業務（0-2pt）→ 即時RPA化可能（Phase 4で計画）
準判断業務（3-4pt）→ 2026年内にパイロット、2027年本番移行
判断業務（5-6pt）→ 2026年はAI情報提示のみ、移行は再評価後
高度判断業務（7-9pt）→ 技術的変化がない限り人間必須を維持
```

> **注**: #14、#15は3-4点の準判断業務だが、5つの客観テスト③④により「人間最終承認必須」と判定。完全自動化対象から除外する。

### 15項目承認の意義

```
役員会での承認内容:
  「以下の15業務は人間が判断・責任を持つ業務として定義し、
   AI/RPAによる代替を行わない（または最終承認は人間が行う）ことを確認する」

この承認により確定すること:
  ①  残りの16業務（実行業務10件＋準判断業務6件から境界値2件除外）が自動化対象として確定
  ② Phase 5の人員削減シミュレーションの論理基盤が確定
  ③ 「AIが判断業務を奪う」という組織的誤解への公式な回答が完成
  ④ 9月20日最終承認への論理的連続性が確保される
```
