> **このフェーズの要点（役員会向け3行）**

> - 5月20日時点での決議事項は「人員削減承認」ではなく「**判断業務15項目の定義承認**」。削減の根拠を先に固めることが最大のリスク回避
> - 年間4,900万円の人件費削減に対し初期投資3,000〜5,000万円、ペイバック約1年。3年累計ネット効果は**約1億円**

## 1-1. 結論：3つのKey Message

### Key Message 1: 削減可能工数は年間約11,500時間（7名相当）

業務改革部26名の年間総工数（約43,680時間）のうち、AI/RPA化・外注化・業務廃止により削減可能な工数は**約11,500時間**。これは7名の年間標準稼働時間（1,680h/名×7=11,760h）にほぼ合致する。

### Key Message 2: 削減の82%はAI/RPA+業務廃止で代替可能

| 代替手段 | 削減工数 | 割合 |
|---------|:-------:|:---:|
| RPA完全自動化 | 3,200h | 28% |
| 生成AI+RPA（Human-in-Loop） | 5,100h | 44% |
| 一部外注化 | 1,200h | 10% |
| 業務廃止・標準化 | 2,000h | 17% |
| **合計** | **11,500h** | **100%** |

### Key Message 3: 判断業務として保護すべき業務は全体の約30%

AI・RPA化しない（すべき）「判断業務」は業務全体の約30%（チーム別：生産計画35%・物流25%・受注10%・業務総括60%）。これを先に定義することで、「AIで全員削減」という誤解を組織的に解消できる。

## 1-6. 本プロジェクトのReframing（最重要前提）

> **本プロジェクトは「人員削減プロジェクト」ではなく「業務再定義プロジェクト」である**

このReframingが成功の鍵。理由：

| 旧フレーム（危険） | 新フレーム（本プロジェクト） |
|---|---|
| 削減数（▲7名）から逆算して業務を切る | 業務を定義してから削減数が導き出される |
| AI化できるものをできるだけ多く | 人間が必須な業務を先に守る |
| 現場は削減される側 | 現場は業務定義の主体者 |
| 結論先行で役員会承認 | 根拠先行で役員会承認 |

---

> **エグゼクティブサマリー総括**: 太陽油脂業務改革部SCM領域の▲7名（26名→19名）は、「判断業務の客観的定義」→「AI/RPA適用可否判定」→「工数ボトムアップ積み上げ」の順序で論理的に導き出される。5月20日の最重要成果物は人員削減計画ではなく「判断業務定義文書」であり、これを役員会で承認することが、9月20日最終承認への確実な論理基盤となる。

## 2-5. 太陽油脂固有の事業特性（SCM複雑性の源泉）

### 脂肪酸・油脂化学品製造の特性

| 特性 | 内容 | SCM業務への影響 |
|---|---|---|
| **バッチ生産** | 連続生産ではなくバッチ単位で製造 | 日程計画に反応条件・清掃時間の制約が介在。単純最適化ソルバーでは対応不可 |
| **連産品・副産物** | 脂肪酸精製時にグリセリン等が副産物として発生 | 主産物と副産物の生産配分は市況・契約を複合判断。典型的高度判断業務 |
| **原料油脂の市況変動** | ヤシ・パーム・牛脂等の価格が市況連動 | 調達タイミング・量判断に相場読みが必要 → 完全自動化困難 |
| **化粧品原料GMP** | 化粧品原料は品質保証基準が厳格 | 受注・生産・物流に品質判断が介在。法令上の説明責任が発生 |

### 石鹸・OEM化粧品事業の特性

| 特性 | 内容 | SCM業務への影響 |
|---|---|---|
| **少量多品種** | OEM案件を含む多品種構成 | 受注の例外処理比率が製造業平均より高い。標準化の限界 |
| **ロット仕様指定** | 顧客ごとに成分・規格が異なる | FAX/メール受注の単純OCR化では対応不可。仕様解釈が必要 |
| **長期BtoB契約** | 大手化粧品メーカー等との長期取引 | 需要予測にAI活用余地あるが、契約変更は人間判断必須 |

### 親会社物流ユニットとの連携特性

- **物流チームの一部業務が親会社物流ユニットと不可分に連携**している

- 親会社側の同意なしに物流削減は実行不可能 → Phase 6のSH管理で最優先事項

## 3-1. フレームワーク設計思想：なぜ「定義」が先か

### 本プロジェクトが犯してはならない思考順序

```
【危険な思考順序（やってはいけない）】
  削減目標（▲7名）→ 削減できる業務を探す → 判断業務を後付けで守る

【正しい思考順序（本フレームワーク）】
  判断業務を客観的に定義・保護 → 残余業務の自動化可否を判定 → 削減工数が積み上がる → 人員数が導き出される
```

この順序を守ることで、「AIで全員削減するのでは」という現場の誤解を解消し、役員会での論理的な承認を可能にする。

### 3軸評価モデルの設計原則

| 設計原則 | 内容 |
|---|---|
| **客観性** | 感覚的な「重要業務」を排除し、3軸の点数で分類する |
| **監査可能性** | 将来の労務問題に備え、分類根拠を文書化・保存する |
| **変動対応** | 技術進化により分類が変わりうることを前提に、再評価プロセスを組み込む |
| **現場合意** | 評価は管理職だけでなく担当者ヒアリングを経て確定する |

## 3-2. 3軸スコアリングモデル

### 軸の定義と採点基準

#### 軸A: 情報完全性（Completeness of Information）
> 「この業務の判断に必要な情報は、構造化データとして100%揃っているか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | 必要情報がすべて構造化データ（DB/CSV）で存在 | 標準品の在庫残高確認、定型受注の価格参照 |
| **1点** | 主要情報は構造化、一部は文書/口頭情報を参照 | 通常の生産計画（BOMあるが一部現場確認要） |
| **2点** | 構造化データ5割＋非構造化情報（メール・商談内容）5割 | MC品目の需要予測（市場感・顧客ヒアリング込み） |
| **3点** | 市場感・関係性・暗黙知が判断の主要情報源 | 不採算品の製造継続判断、新原料調達先選定 |

#### 軸B: 判断再現性（Reproducibility of Decision）
> 「同じ情報を与えれば、担当者が誰でも同じ判断をするか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | ルール・閾値が完全定義されており誰でも同一結果 | 安全在庫下回りアラート発報、定型受注入力 |
| **1点** | 大半のケースはルール適用、例外時は判断が分かれうる | 標準品MRP実行、通常配車割当 |
| **2点** | ルールはあるが経験・文脈で判断が変わる（±20%程度） | 需給逼迫時の優先順位付け、クレーム対応方針 |
| **3点** | 担当者の経験・判断が本質、標準化不可 | 共同製品の市場価値判断、新規顧客の与信判断 |

#### 軸C: 例外発生頻度（Frequency of Exceptions）
> 「想定外の状況・判断への割り込みはどのくらい発生するか？」

| 点数 | 定義 | 太陽油脂における具体例 |
|:---:|---|---|
| **0点** | 例外はほぼゼロ（月1件未満） | 週次定型レポート出力、FAX受注の定型入力 |
| **1点** | 例外は月数件、ルールで対処可能 | 原料価格変動時の発注量調整、返品処理 |
| **2点** | 週次で例外が発生し、都度判断が必要 | バッチ製造の歩留まり異常対応、配送遅延調整 |
| **3点** | 日常的に想定外が発生し、即時判断が事業継続に直結 | 原料タイト時の製造優先品目決定、大口解約対応 |

### 3軸合計スコアによる4分類

```
スコア計算式: 合計点 = 軸A + 軸B + 軸C （最大9点）

  0〜2点 ┌─────────────────────────────────────────────┐
         │  【実行業務】                                  │
         │  AI/RPA完全自動化対象。人間の介入なしで処理可能  │
         │  → Phase 4でRPA/OCR化計画を策定               │
         └─────────────────────────────────────────────┘

  3〜4点 ┌─────────────────────────────────────────────┐
         │  【準判断業務】                                │
         │  AI提案＋人間最終承認（Human-in-Loop）         │
         │  → Phase 4で生成AI+承認フロー設計              │
         └─────────────────────────────────────────────┘

  5〜6点 ┌─────────────────────────────────────────────┐
         │  【判断業務】                                  │
         │  人間主体、AI情報提示支援のみ許容              │
         │  → 5/20役員会の承認対象（保護業務として確定）   │
         └─────────────────────────────────────────────┘

  7〜9点 ┌─────────────────────────────────────────────┐
         │  【高度判断業務】                              │
         │  人間必須。AI代替不可・外注不可               │
         │  → 5/20役員会の承認対象（絶対保護）            │
         └─────────────────────────────────────────────┘
```

## 3-3. 5つの客観テスト

スコアリングの妥当性を検証するための追加テスト。スコアが3〜4点（境界値）の業務に適用する。

| テスト | 問い | 判断業務と判定する条件 |

| **②説明可能性テスト** | 判断根拠を第三者に論理的に説明できるか（Yes/No以外）？ | 説明に「感覚・経験」が必要な場合 → 判断業務 |

| **④誤り事後検証テスト** | AI判断が誤った場合、事業損害が発生するまでの検出猶予は24h以上あるか？ | 24h未満の場合 → 判断業務（人間監視必須） |
| **⑤代替可能性テスト** | その担当者が突然不在になった場合、別の担当者が同等の判断を即座にできるか？ | できない場合 → 高度判断業務 |

## 3-4. AI/RPA化判定ルール（4条）

### Rule 1: 強制人間判断条件（いずれか該当でAI代替禁止）
- [ ] 判断が法令・コンプライアンス上の責任を伴う
- [ ] 判断が従業員の雇用・評価に直接影響する
- [ ] 誤判断の損害額が単月売上の1%超になる可能性がある
- [ ] 顧客・サプライヤーとの関係維持に「人の判断」が期待されている

### Rule 2: AI支援許容条件（準判断業務の要件）
- [ ] AI出力を人間が最終確認・承認するフローが設計されている
- [ ] AI判断の根拠（説明性）が担当者に提示される
- [ ] 誤判断時のロールバック手順が定義されている
- [ ] 月次でAI精度をレビューする担当者が指名されている

### Rule 3: 完全自動化条件（実行業務の要件）
- [ ] 処理対象データがすべてデジタル・構造化されている
- [ ] 判断ロジックがIf-Then-Else形式で完全記述できる
- [ ] 例外発生時に人間へのエスカレーションルートが自動発動する
- [ ] 月次の自動化精度が98%以上を維持できることが確認されている

### Rule 4: 段階的移行ルール
```
実行業務（0-2pt）→ 即時RPA化可能（Phase 4で計画）
準判断業務（3-4pt）→ 2026年内にパイロット、2027年本番移行
判断業務（5-6pt）→ 2026年はAI情報提示のみ、移行は再評価後
高度判断業務（7-9pt）→ 技術的変化がない限り人間必須を維持
```


### ヒアリング設計方針

| 項目 | 内容 |
|---|---|
| 対象 | 業務改革部SCM領域 26名（全員） |
| 形式 | チーム別グループヒアリング（4チーム×2回） + 個別確認（必要に応じて） |
| 時間 | 90分/回 |
| 実施者 | アドバイザー + 業務改革部長（同席） |
| 目的 | 業務スコアリングの現場検証・異議収集・合意形成 |

| 日程 | チーム | 参加者 | アジェンダ |

| 5月9日（金）10:00〜11:30 | 物流チーム | チーム長+8名全員 | B-01〜B-09スコア確認 |
| 5月12日（月）13:00〜14:30 | 受注チーム | チーム長+7名全員 | J-01〜J-08スコア確認 |
| 5月13日（火）10:00〜11:30 | 業務総括チーム | チーム長+4名 | S-01〜S-06スコア確認 |
| 5月15日（木）14:00〜15:30 | 全チーム長 | 4名 | 最終スコア確認・異議処理 |
