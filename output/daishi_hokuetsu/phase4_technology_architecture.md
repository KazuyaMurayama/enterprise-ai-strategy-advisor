# フェーズ4: 技術アーキテクチャ戦略 — 第四北越銀行

> 作成日: 2026年4月14日

---

> **このセクションの要点**
> - 第四北越銀行のAIアーキテクチャは「金融庁規制対応必須の融資AI・AML AI領域」「顧客・渉外向けビジネスAI」「行内業務AI」の3層分離設計が必要。勘定系への直接接続は禁止し、安全なAPIブリッジを経由させる
> - 最大の技術課題は「老朽化した勘定系システム（NTTデータのFINEMAX等）とのAPI連携」。2007年合併後のシステム統合で異なる世代のシステムが混在しており、APIブリッジ設計が最優先技術課題
> - 基盤モデルはClaude Sonnet 4.6（渉外AI・コールセンターAI・行内AI）とAzure OpenAIを主軸に、融資スコアリング・AML検知は専門の金融AIベンダーとのPartner戦略が最適

---

## 4-1. 銀行AI 3層アーキテクチャ（第四北越銀行版）

```
┌───────────────────────────────────────────────────────────────┐
│  Layer A: 行内業務AI（非規制対象）                               │
│  [渉外担当AIアシスタント] [ナレッジ管理AI] [書類OCR AI]          │
│  [コールセンターAI] [競合情報AI] [HR AI] [財務報告AI]            │
│  ↑ Claude Sonnet 4.6 API / Azure OpenAI + RAG基盤             │
├───────────────────────────────────────────────────────────────┤
│  Layer B: ビジネスコアAI（金融庁届出・監督下）                    │
│  [融資審査AIスコアリング] [詐欺検知AI] [解約予測AI]              │
│  [資産運用AIアドバイザー] [不動産担保評価AI]                     │
│  ↑ バリデーション済み金融AIプラットフォーム                      │
│  ↑ 金融庁システムリスク管理規制対応・内部統制文書化              │
├───────────────────────────────────────────────────────────────┤
│  Layer C: AML/KYC AI（最高規制強度）                            │
│  [疑義取引検知AI] [顧客リスクプロファイルAI] [STR自動生成AI]     │
│  ↑ 犯収法・金融庁AMLガイドライン対応 + 当局報告フロー統合        │
│  ↑ 完全な監査ログ・説明可能性（XAI）必須                        │
└───────────────────────────────────────────────────────────────┘
            ↕  銀行AI中間層API（勘定系との安全なブリッジ）
            ↕  勘定系 / 融資DB / CRM / ATMネットワーク
            ↕  Zero Trust セキュリティ基盤
```

## 4-2. 勘定系レガシーとの統合戦略

| 連携対象システム | 技術課題 | 対応方針 | 期間・コスト |
|--------------|--------|---------|:----------:|
| 勘定系（FINEMAXⅡ等） | APIなし・バッチ処理中心 | API Gatewayによる安全なラッパー構築。読取専用APIを優先 | 6-9ヶ月 / 1.2億円 |
| 融資管理DB | レガシーSQL・連携困難 | 日次バッチでDWH（データウェアハウス）に同期→AIが参照 | 3-4ヶ月 / 0.5億円 |
| 渉外・法人CRM（SalesForce等） | AI連携API一部あり | 既存API活用＋AIアシスタントのWebhook統合 | 2-3ヶ月 / 0.3億円 |
| AMLシステム（既存） | 旧世代ルールベース | AI層を既存システムに重ねて追加（ラッパー型） | 4-6ヶ月 / 0.8億円 |

## 4-5. AIセキュリティ10領域フレームワーク（Phase 4担当：領域2・5・7・9）

### 領域2：AIサプライチェーンセキュリティ
| 対策 | 第四北越への適用 | 優先度 |
|-----|------------|:-----:|
| 外部LLMサービス認証確認 | Claude/Azure OpenAI利用前にSOC2 Type II・金融機関向け条項確認。年次更新 | ★★★★★ |
| AML AIベンダーの認証 | NICE Actimize等のAML AIの金融庁適合性確認。SLA・データ取扱条件を精査 | ★★★★★ |
| OSS依存関係スキャン | 渉外AIのOSSにSnyk適用。顧客情報漏洩リスクの依存関係監視 | ★★★★☆ |

### 領域5：プライバシー保護技術（PETs）
| 技術 | 銀行業での適用 | 実装 |
|-----|------------|-----|
| 差分プライバシー | 融資AIの学習データ（顧客財務情報）への適用 | Layer B AIモデル学習段階に必須適用 |
| k-匿名化（k≥10） | 顧客属性データを渉外AIが参照する際の匿名化 | CRMデータのAI連携時に必須 |
| トークン化 | 口座番号・マイナンバーのAIシステムへの直接入力禁止 | Layer A/B/C全体に適用 |

### 領域7：データリネージ
| 管理対象 | 要件 | ツール |
|---------|-----|:----:|
| 融資AI学習データ | 顧客財務データ → クリーニング → 特徴量変換 → 学習データまでの全ステップ | Apache Atlas |
| AML AI判断ログ | 取引データ → 前処理 → AI推論 → 当局報告までの完全リネージ | カスタムリネージDB |
| LLM参照文書（RAG） | 渉外AIが参照した規程・商品情報のバージョン記録 | LangSmith |

### 領域9：Zero Trust for AI
| 原則 | 適用 | 実装 |
|-----|-----|-----|
| 最小権限 | Layer A（渉外AI）とLayer C（AML AI）のAPIキーを完全分離 | Azure RBAC + API Management |
| 継続的認証 | 渉外担当のAI利用は生体認証による継続確認 | Azure AD + 生体認証 |
| マイクロセグメンテーション | 勘定系DBへのアクセスはLayer B/Cのみ。Layer Aから直接接続禁止 | Azure VNet + NSG |
| AI入出力検査 | 全AIへの入出力をリアルタイムスキャン。顧客情報・口座情報の外部送信を自動ブロック | Azure AI Content Safety + DLP |
