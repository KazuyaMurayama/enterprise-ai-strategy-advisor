# フェーズ4: 技術アーキテクチャ戦略 — Amazonジャパン合同会社

> 作成日: 2026年4月20日

---

> **このセクションの要点**
> - AmazonジャパンのAIアーキテクチャは「グローバルAI基盤の活用」と「日本固有のカスタマイズ層」を明確に分離する2層構造が最適。グローバルAI（推薦・物流・AWS）の恩恵を最大化しつつ、日本語・日本法規・日本文化への対応を独自レイヤーで実装する
> - 基盤モデルはClaude Sonnet 4.6（カスタマーサービス・EC推薦補助・AWS Bedrock提供）を主軸に、物流最適化・需要予測は専用Transformerモデル。「日本語品質」が全てのAIの差別化軸
> - データ戦略の核心：AmazonのEC・Prime・広告・物流の4大データストリームを日本版統合データプラットフォームで横断活用。他のECプレイヤーが持ち得ない購買×広告×物流×コンテンツの統合が最強の競合優位

## 4-3. Build / Buy / Partner 判断マトリクス

| ユースケース | 判断 | 理由 |
|-----------|:---:|-----|
| EC推薦エンジン（コア） | **Build**（グローバル流用+日本カスタム） | Amazonの最重要競合優位。外部依存は不可 |
| 日本語LLM | **Partner（Anthropic Claude）+ Build** | Claude Sonnet 4.6 APIをBedrock経由で活用。日本語ファインチューニングは自社 |
| 物流ルーティングAI | **Build**（グローバルエンジン+日本モジュール） | 日本の住所・物流網の固有性が高く外部に任せられない |
| CS AIエージェント | **Buy+Customize**（Claude API + Amazon Connect） | 短期立ち上げを優先。Claude品質が最高水準 |
| Amazon Advertising AI | **Build**（グローバル基盤+日本特化） | 購買データ × 広告の掛け合わせは自社資産 |
| ドローン配送（将来） | **Build+Partner**（Prime Air + 国内航空規制対応） | 独自技術+日本規制対応パートナーが必要 |
| 倉庫ロボット（FC） | **Build**（Amazon Robotics流用） | Kiva/Amazonian Roboticsのグローバル投資を流用 |

    EC推薦AI      物流最適化AI    広告ターゲAI    AWS顧客インサイトAI

### データガバナンス（日本版）

| データ種別 | 個人情報保護法分類 | 管理方針 |
|---------|:-------------:|--------|
| Prime会員購買履歴 | 個人情報（要同意） | 匿名化処理後AIモデル学習。個人特定可能データはAPPI準拠管理 |
| 配送先住所・時間情報 | 個人情報 | 推薦・物流最適化にのみ利用。第三者提供禁止 |
| 広告クリックデータ | 行動情報（準個人情報） | オプトアウト選択肢の明示。匿名化コホートでのみ外部利用 |
| AWS利用ログ（企業） | 非個人情報 | モデル改善利用は契約上明示。機密データは利用禁止 |

## 4-5. AIセキュリティ10領域（Phase 4担当：領域2・5・7・9）

### 領域2：AIサプライチェーンセキュリティ

| リスク | Amazonでの具体例 | 対策 |
|-------|-------------|-----|
| **基盤モデルのバックドア** | Claude API・Titan APIへの依存リスク | Amazon Bedrockの自社インフラ内完結。モデル出力の定期的な安全性監査 |
| **サードパーティAIライブラリ** | PyTorch/TensorFlow CVEの脆弱性 | Amazon CodeGuru + 毎月SBOMスキャン（Snyk統合） |
| **出品者・広告主の悪意あるAIデータ** | 偽レビュー・虚偽商品情報でAIを汚染 | Amazon Project Zero（ブランド保護AI）+ 継続的データ品質監視 |

### 領域5：プライバシー強化技術（PET）

| 技術 | 適用シナリオ |
|-----|-----------|
| **差分プライバシー** | Prime会員行動データをモデル学習に使用する際の個人識別防止 |
| **連合学習** | デバイス（Alexa）上での音声データ学習（データをクラウドに送らない） |
| **同形暗号** | 医療データ（Amazon Health）の暗号化状態でのAI分析 |
| **データマスキング** | CS対応AIが個人情報を扱う際の自動マスキング |

### 領域7：データ系譜（Data Lineage）

```
[EC購買ログ収集] → [前処理・匿名化] → [モデル学習] → [推薦AI出力]
    ↓                    ↓                ↓              ↓
 収集元記録            APPI準拠確認       モデルバージョン   アウトプット記録
 タイムスタンプ         同意確認           学習データ履歴     根拠ログ
 
実装: AWS Glue DataBrew + Amazon SageMaker Lineage Tracking
JFTC対応: アルゴリズム透明性レポート用に推薦ロジックの説明可能な系譜を保管
```

### 領域9：AIゼロトラスト

| 原則 | 実装 |
|-----|-----|
| **デバイス認証** | AWS IAM Identity Center + 多要素認証（倉庫端末・配送車両デバイス） |
| **最小権限** | AI推論APIへのアクセスはロール別スコープ制限。モデル学習環境と推論環境を分離 |
| **継続監視** | Amazon GuardDuty + AWS Security Hub（24時間AI推論API監視） |
| **暗号化** | KMS（AWS Key Management Service）による全データ暗号化。日本リージョン内完結 |
