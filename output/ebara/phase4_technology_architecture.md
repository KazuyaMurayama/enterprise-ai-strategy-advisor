# フェーズ4: 技術アーキテクチャ戦略 — 株式会社荏原製作所

> 作成日: 2026年4月14日

---

> **このセクションの要点**
> - 荏原のAIアーキテクチャは「半導体CMP装置（最高機密層）」「産業IoT・フィールドサービス（規制・IP保護層）」「業務・社内AI（標準層）」の3層構成。CMP装置の顧客プロセスデータは最高水準の分離・保護が必須
> - Build/Buy/Partnerの原則：CMP装置AIとポンプ予知保全AIは競合優位の核心なのでBuild主体。設計支援AI（CAE×生成AI）はClaude API活用のBuy+Customize。汎用業務AIはSaaS活用
> - 基盤モデル選定：設計支援・技術文書AIにはClaude Sonnet 4.6（API）、複雑な工学文書・CMP最適化解析にはClaude Opus 4.6。産業IoT予知保全はTime-series特化モデル（Prophet/Transformer）

---

## 4-1. 全体AIアーキテクチャ（3層構成）

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer C: 半導体CMP装置AI層（最高機密・完全分離）                    │
│  ・CMP装置プロセスデータ（顧客IP）                                   │
│  ・TSMC/Samsung等の工場内センサーデータ                              │
│  ・プロセスレシピ（機密）                                            │
│  ─── エアギャップ/専用回線のみ ──────────────────────────────────  │
│  Layer B: 産業IoT・フィールドサービスAI層（規制・IP保護）             │
│  ・ポンプ・圧縮機IoTデータ（50万台）                                  │
│  ・工場製造ラインデータ                                              │
│  ・焼却炉・環境プラントデータ                                          │
│  ─── VPN+TLS+ゼロトラスト ────────────────────────────────────  │
│  Layer A: 業務・社内AI層（標準セキュリティ）                          │
│  ・設計支援AI（CAE×生成AI）                                         │
│  ・技術文書AI・コールセンターAI                                       │
│  ・社内ナレッジAI・HR/財務AI                                         │
└─────────────────────────────────────────────────────────────────┘
```

## 4-2. Layer別 技術スタック詳細

| コンポーネント | 技術選択 | 選定理由 |
|------------|:------:|--------|

### Layer B：産業IoT・フィールドサービスAI層

| コンポーネント | 技術選択 | 選定理由 |

| **エッジAI** | ONNX Runtime on ARM（現場設置機器） | 通信途絶時も予測継続 |
| **製造品質AI** | Computer Vision（品質検査）+ プロセスAI | 製造ラインのリアルタイム制御 |
| **焼却炉AI** | 強化学習（燃焼制御最適化）+ Build | 環境規制対応の差別化 |

### Layer C：半導体CMP装置AI層

| コンポーネント | 技術選択 | 選定理由 |
|------------|:------:|--------|
| **CMPプロセスAI** | 専用MLパイプライン（Build、クローズド環境） | 顧客IP保護のため外部クラウド禁止 |
| **エンドポイント検出AI** | 時系列分類モデル（LSTM）＋顧客共同開発 | 装置ごとのカスタマイズが必要 |

| **データ管理** | 暗号化ローカルストレージ + アクセスログ完全記録 | 顧客データの厳格管理 |

## 4-4. データ戦略

### データカタログ（荏原主要データ資産）

| データ種別 | 量・規模 | AI活用機会 | Layer | 管理方針 |
|---------|:-------:|----------|:-----:|--------|

### データパイプライン（Layer B: IoT→AI）

```
現場センサー（IoTデバイス）
    ↓ MQTT/OPC-UA
エッジゲートウェイ（ONNX推論+異常検知）
    ↓ TLS 1.3暗号化通信
Azure IoT Hub（データインジェスト）
    ↓ Stream Analytics（リアルタイム処理）
Azure Data Lake Storage Gen2
    ↓ Azure Databricks（ML学習・バッチ処理）
予知保全AIモデル（PyTorch/MLflow管理）
    ↓ REST API
フィールドサービスダッシュボード（Power BI + カスタムUI）
```

## 4-5. AIセキュリティ10領域（Phase 4担当：領域2・5・7・9）

### 領域2：AIサプライチェーンセキュリティ

| リスク | 荏原での具体例 | 対策 |
|-------|-------------|-----|
| **基盤モデルのバックドア** | Claude API経由の設計データ流出リスク | PII/機密データをAPIに送信しないプロンプト設計 |
| **OSS MLライブラリの脆弱性** | PyTorch/scikit-learnの脆弱CVE | 毎月SBOMスキャン（Snyk/Trivy） |
| **クラウドプロバイダーリスク** | Azure依存による障害・サービス終了 | マルチクラウド戦略（Azure主軸+AWS補完） |
| **IoTデバイスファームウェア** | エッジAIデバイスへの不正ファームウェア | セキュアブート+署名付きファームウェア更新のみ許可 |

### 領域5：プライバシー強化技術（PET）

| 技術 | 適用シナリオ | 荏原での実装 |
|-----|-----------|-----------|
| **連合学習（Federated Learning）** | 顧客サイトのCMPデータをサイト外に出さずにAI学習 | 顧客工場内でローカル学習→モデル差分のみ転送 |
| **差分プライバシー** | 複数顧客のポンプデータを統合学習する場合 | ε=1.0設定でデータ識別不可能性を保証 |
| **秘密計算（MPC）** | 競合メーカー間でのベンチマークデータ共有 | 将来の業界AI連合構想での適用 |
| **データマスキング** | 顧客名・設置先情報をAI学習データから除去 | パイプライン上での自動匿名化処理 |

### 領域7：データ系譜（Data Lineage）

```
データ系譜追跡の対象フロー:
[生データ取得] → [前処理・特徴量エンジニアリング] → [AIモデル学習]
     ↓                    ↓                            ↓
  取得元記録           変換ロジック記録               学習データバージョン記録
  取得時刻記録         品質チェック結果記録            モデルバージョン記録
  
実装: Apache Atlas + Azure Purview
保管: データカタログ（最低5年保管）
目的: AI判断の根拠遡及・規制対応・品質保証
```

| 適用領域 | 系譜の重要性 | 実装優先度 |
|--------|-----------|:--------:|
| CMP装置AI | 顧客歩留まり判断根拠の遡及 | 最高 |
| 製造品質AI | 製品不良原因トレーサビリティ | 最高 |
| 予知保全AI | 保全判断根拠の法的証明 | 高 |
| 焼却炉AI | 環境規制対応の記録義務 | 高 |

### 領域9：AIゼロトラスト

```
ゼロトラスト実装アーキテクチャ:

外部（顧客サイト/フィールド）
    ↓
[ID/デバイス認証] Microsoft Entra ID + CBA証明書認証
    ↓
[条件付きアクセス] デバイス準拠+場所+リスクスコア
    ↓
[マイクロセグメンテーション] Layer C / B / A 間の厳格分離
    ↓
[最小権限] ロールベースアクセス（RBAC）+ 時間制限アクセス
    ↓
[継続監視] Microsoft Sentinel + Defender for Cloud
    ↓
AI推論API（認可済みリクエストのみ）
```

| ゼロトラスト原則 | Layer C適用 | Layer B適用 | Layer A適用 |
|--------------|:---------:|:---------:|:---------:|
| デバイス認証 | 専用証明書 | IoT Hub証明書 | Entra ID |
| 最小権限 | 装置毎分離 | 工場毎分離 | 部門毎分離 |
| 継続監視 | 24/365専任 | SOC監視 | 標準監視 |
| 暗号化 | E2E+保存暗号化 | TLS+保存 | TLS |
