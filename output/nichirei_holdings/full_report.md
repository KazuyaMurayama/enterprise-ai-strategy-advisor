# 2. AS-IS診断（現状分析）

## 2-1. AS-IS診断の4軸フレーム

当社の現状を **業務 × データ × システム × 組織** の4軸で診断した。

| 軸 | 現状 | 主たる痛点 | 影響額（年間試算） |
|---|---|---|---|
| **業務** | 各部署が類似帳票をExcelで個別作成。月次決算後の業績把握に2-3週間 | 帳票属人化・数値微差・経営判断遅延 | △10億円 |
| **データ** | グループ各社・部門間でデータ定義が不統一。マスタが分散 | データ品質低・名寄せ困難・信頼性疑義 | △8億円 |
| **システム** | SAP（基幹）→DIVA（連結会計）への月次バッチ連携。BIツールが3種混在 | リアルタイム性ゼロ・ツール乱立・保守費過大 | △12億円 |
| **組織** | データガバナンス担当不在。情報戦略部と各事業会社IT部門が並列 | 全社最適視点欠如・横串の意思決定機能不全 | △5億円 |
| **小計（年間ロス）** | — | — | **△35億円** |

## 2-2. システム現状マップ（テキスト構造）

```
[Layer 5: BI] 持株会社: Power BI（一部稼働） / ニチレイフーズ: Tableau / ニチレイロジ: MotionBoard
                                  ↑（個別接続・データ重複）
[Layer 4: DWH/データマート] 旧オンプレDWH（保守費高） + Snowflake（日立F&L移行中、Y1完了予定）
                                  ↑（CDC未実装、バッチ連携）
[Layer 3: 連結・管理会計] DIVA（連結会計、月次連携）
                                  ↑（月次バッチ）
[Layer 2: ERP/基幹] SAP（持株会社・主要事業会社）+ 個別基幹（一部事業会社）
                                  ↑
[Layer 1: 業務システム] 各事業会社（製造実行、物流、営業、調達等）多数
```

### 主な技術的痛点
1. **SAP→DIVA連携が月次**: 業績の週次・日次把握不能。経営層が前月数値で議論せざるを得ない
2. **マスタデータ不統一**: 取引先コード・商品コード・組織コードがグループ各社で別管理
3. **BIツール乱立**: ライセンス費年間2-3億円超、保守体制分散、ナレッジ共有不能
4. **データカタログ不在**: 「どこに何のデータがあるか」を尋ねる先がない

# 3. TO-BE像（目指す姿）

## 3-1. データドリブン経営の定義（当社版）

当社が目指す「データドリブン経営」を次の5要件で定義する：

| # | 要件 | 現状 | 5年後TO-BE |
|---|---|---|---|
| 1 | **業績の日次把握** | 月次（DIVA経由） | **翌日朝7時に前日業績がダッシュボード表示** |
| 2 | **単一のデータソース（SSoT）** | 各部署バラバラ | **持株会社CoE管理のSnowflake DWHが唯一の正** |
| 3 | **セルフサービスBI** | IT部門依頼ベース | **業務部門が自らPower BIで分析・帳票作成** |
| 4 | **データ品質の保証** | 品質定義なし | **Data Steward制度によりKPIごとに品質基準・責任者明示** |
| 5 | **データ駆動意思決定** | 経験・勘ベース | **経営会議の主要議題は全てダッシュボード起点** |

## 3-2. TO-BE業務イメージ（経営会議の景色）

### As-Is（現状）
- 月次決算後3週間で前月業績資料が完成
- 経営会議で「先月の数値」を議論
- 重要KPIは Excel で各部署が手作りした資料
- 数値の微差で議論が脱線（「どの数字が正しいのか」）

### To-Be（3年後）
- **毎朝7時、前日までの業績・KPIがPower BIダッシュボードで自動更新**
- 経営会議は **ダッシュボード起点** で議論。資料は議論補助のみ
- KPI定義はCoEのDataカタログで管理、定義の議論不要
- アラート機能で異常値検知 → 経営層スマホへ即時通知

## 3-3. TO-BEデータアーキテクチャ概観（5層モデル）

| Layer | TO-BE構成 | 主要ツール候補 |
|---|---|---|
| L5: 利用層 | グループ統一BI（Power BI主軸）+ 既存BI共存 | Power BI Premium + Tableau/MotionBoard継続 |
| L4: セマンティック層 | KPI定義の単一管理。指標統一 | Power BI Dataset + dbt（Semantic Layer） |
| L3: データウェアハウス | Snowflake（持株会社管理）+ グループ各社マート | Snowflake |
| L2: データ統合 | SAP直接連携（リアルタイム/日次）+ ETL/ELT | SAP Datasphere or Fivetran + dbt |
| L1: ソース | SAP, DIVA, 各事業会社基幹, 物流TMS, IoT等 | （既存） |

# 4. データ基盤アーキテクチャ戦略

## 4-1. 基本設計方針（5原則）

| # | 原則 | 内容 |
|---|---|---|
| 1 | Single Source of Truth | Snowflake DWHを唯一の正とし、業務システムから直接BIを繋がない |
| 2 | セマンティックレイヤー統一 | KPI定義はCoE管理の単一レイヤーに集約 |
| 3 | ベンダーロックイン回避 | OSS（dbt等）と標準SQLを優先。ベンダー固有機能は最小化 |
| 4 | 段階的マイグレーション | ビッグバン回避。並行稼働期間を設けてリスク低減 |
| 5 | Run/Buildコスト分離 | 運用コストと開発コストを別予算化し、KPI管理 |

## 4-2. SAP×Snowflake×Power BI 統合アーキテクチャ

### 連携設計（SAP→Snowflake のリアルタイム化）

| 方式 | リアルタイム性 | コスト | 推奨度 |
|---|:---:|:---:|:---:|
| **A. SAP Datasphere（旧 SAP DWC）** | 高（数分） | 高 | ★★★★☆ |
| **B. Fivetran/Hevo（CDC ベース ELT）** | 中（数分〜数時間） | 中 | ★★★★★ |
| C. 自社開発 ETL（バッチ） | 低（日次） | 低 | ★★★☆☆ |
| D. SAP標準IDoc連携 | 低（日次） | 低 | ★★☆☆☆ |

[推奨] **方式 B（Fivetran/Hevo）** を採用し、管理会計データを Snowflake へ日次（将来は数時間ごと）抽出する。理由: (1) ベンダー中立、(2) 導入スピード、(3) SAP側改修最小、(4) 将来的に他データソース追加が容易。

### Power BI セマンティックレイヤー設計

- **Power BI Premium Capacity** をグループ共有で確保
- **共通データセット**（Power BI Dataset）をCoEが提供。各部署はそれを参照
- **KPIロジック**はdbtでSnowflake上に実装、Power BIは可視化に専念
- **行レベルセキュリティ（RLS）** で事業会社間のデータ分離

## 4-3. マスターデータ管理（MDM）戦略

### 統合対象マスタ（優先順位）

| 優先度 | マスタ | グループ内現状 | 統合方針 |
|:---:|---|---|---|

## 4-6. マルチBI共存戦略（最重要論点）

各事業会社で既に Tableau / MotionBoard が稼働。Power BI への強制統一は現実的でない。

### 共存原則
| BIツール | 用途 | 残置方針 |
|---|---|---|
| **Power BI**（持株会社・新規） | **全社標準。経営層・財務・人事** | 拡大 |
| Tableau（ニチレイフーズ等） | 既存業務継続。新規開発は原則PowerBI | **既存維持、段階的縮小（Y3以降）** |
| MotionBoard（ニチレイロジ等） | 物流現場ダッシュボード | **当面維持、Y3で再評価** |

### 共通レイヤー戦略
- **データソース層（Snowflake）は統一**
- **BIツール層は3共存**（強制統一しない）
- **KPI定義（セマンティック層）はCoEで統一管理し、各BIツールへ配信**

> **So What?**: 「Power BI強制」ではなく「**データソースとKPI定義の統一**」を本丸とする。BIツールはユーザーが選択。これにより既存投資を活かしつつ、データの一元性を担保。

- **期間**: Month 4-12（9ヶ月）
- **投資**: 約6億円（ライセンス + 開発委託 + チェンジマネジメント）
- **効果**: 持株会社内帳票工数 50%削減、年間¥5-8億円
- **KPI**: 主要帳票20本稼働、利用率80%以上、Excel廃止率50%以上

### Quick Win #3: データガバナンス委員会の発足

## 8-2. 日立F&L ベンダーコントロール戦略（最重要）

### 基本方針: 「**発注側設計主権**」の確立

日立F&Lは技術実装パートナーとして優れているが、**ニチレイのデータ戦略を日立F&Lに委ねてはならない**。本グランドデザイン v1.0 が発注側設計主権の根拠文書となる。

### コントロール6原則

| # | 原則 | 具体的アクション |
|:---:|---|---|
| 1 | **要件はニチレイが定義、設計は協業、実装はF&L** | 要件定義書はCoEが作成 |
| 2 | **アーキテクチャの主要決定はニチレイが最終承認** | 設計レビュー権限保持 |
| 3 | **ベンダー固有技術への依存最小化** | OSS（dbt等）と標準SQL優先 |
| 4 | **データオーナーシップはニチレイ保有** | データ・メタデータ・コードの所有権明示 |
| 5 | **第二ベンダー比較見積を定期取得** | 主要発注は3社見積義務化 |
| 6 | **撤退コストを契約で明示** | データ・コード・ドキュメント引渡し条項 |

### 契約条項整備（Phase 0 で実施）

| 条項 | 内容 |
|---|---|
| データ所有権 | 全データ・メタデータ・コードの所有権はニチレイ |
| ソースコード開示 | カスタム開発分の全ソースコード提供義務 |
| ドキュメント納品 | 設計書・運用手順書・データ辞書の納品義務 |
| 引継ぎ義務 | 契約終了時の知識移転・引継ぎ責任 |
| ベンダー切替権 | ニチレイの判断でベンダー切替可、F&Lは妨害禁止 |
| SLA | 稼働率・障害対応時間の数値約束 |
| 監査権 | ニチレイによる定期監査権 |

## 8-3. データセキュリティ・コンプライアンス

### 個人情報保護対応
- 個人情報は Snowflake 上で **データマスキング**（動的マスキング）
- 利用申請ワークフローを Snowflake 標準機能で実装
- 監査ログを Snowflake Audit から自動収集

### アクセス制御
- **行レベルセキュリティ（RLS）**: 事業会社別データ分離
- **列レベルセキュリティ（CLS）**: 機密項目（個人情報・人件費等）のマスキング
- **役職別アクセス**: Azure AD連携でロールベースアクセス制御

### バックアップ・BCP
- Snowflake標準のTime Travel（90日）+ Fail-safe（7日）
- 主要データは別リージョン定期コピー
- RPO 1時間、RTO 4時間を目標

## 8-4. グループ内データ流通ガバナンス

| 観点 | 方針 |
|---|---|
| グループ内データ利用契約 | 持株会社→事業会社、事業会社間の利用許諾を明文化 |
| 個別事業会社データの独立性 | 競合関連情報（事業会社別利益率等）はアクセス制限 |
| 海外グループ会社 | 個別判断（GDPR等の各国法対応） |

> **So What?（経営層への問いかけ）**
>
> 当社の真の競争力は、冷凍食品No.1のブランドと国内最大の低温物流ネットワークが日々生み出す **データそのもの** にある。しかし、現状ではそのデータの90%以上が活かされていない。
>
> 本グランドデザインは「**AIを導入する前に、データの土台を整える**」という地味だが本質的な投資判断を求めている。AI活用は別途進めるとしても、その効果はデータ基盤の品質を超えない。先にデータを整える企業が、AIの恩恵を最も受ける。
>
> 60億円の投資判断は3年で108%のROIを生む。だが、より重要なのは **「これからの20年間、データを使って勝ち続ける土台」** を作ることである。本書 v1.0 の経営承認をもって、当社のデータドリブン経営の第一歩を踏み出されたい。
