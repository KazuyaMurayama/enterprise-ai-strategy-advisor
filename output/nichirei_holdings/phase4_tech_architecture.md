# フェーズ4: 技術アーキテクチャ戦略 — ニチレイホールディングス AI戦略レポート

---

## このセクションの要点

1. **Snowflake を唯一のデータ統合ハブ**に据え、SAP・DIVA・工場IoT・冷凍倉庫センサーを5層アーキテクチャで統合することで、月次連携の日次化と帳票属人化の構造的解消を実現する。
2. **基盤モデルはClaude 3.7 Sonnetを主軸**に、食品安全・物流規制領域で高い日本語精度と厳格なデータ残留制御を確保しつつ、コストと品質のバランスを最適化する。
3. **技術投資110億円（3年）**はSnowflakeクラウド基盤・AI/ML基盤・Power BI統合・SAP統合・IoT/センサーの5カテゴリに配分し、セキュリティ・ガバナンス領域に総投資の9%（約10億円）を独立計上する。

## 4-1. 技術戦略の全体方針

### Build/Buy/Partner 判定マトリクス

ニチレイのコア競争優位は「冷凍食品の製造ノウハウ」「低温物流ネットワーク（全国370拠点）」「鮮度・食品安全管理データ」であり、これらに直接紐づく機能のみBuildを選択する。

| 機能カテゴリ | 判定 | 根拠 | 対象ベンダー/技術 |
|:---:|:---:|---|---|
| 需要予測AIモデル（冷凍食品特化） | **Build** | 品番・鮮度・気温・イベントの独自特徴量が競争優位源泉 | Azure ML / Snowpark ML |
| 冷凍倉庫温度異常検知AI | **Build** | 自社倉庫センサーデータの独占性が高い | SageMaker / Azure ML |
| 食品トレーサビリティAI | **Build** | 食品衛生法・HACCP対応が独自要件 | 自社開発 + LangChain |
| LLMコパイロット（汎用RAG） | **Buy** | 汎用技術。コモディティ化が速い | Claude API / Azure OpenAI |
| MLOpsプラットフォーム | **Buy** | 差別化なし。OSS流用で十分 | MLflow / Vertex AI |
| データ統合基盤（DWH） | **Partner** | Snowflake＋日立F&L構築。インフラ専門性が必要 | Snowflake + 日立F&L |
| Power BI セマンティック層 | **Partner** | Microsoft パートナー経由で実装 | Microsoft / 日立F&L |
| 帳票自動生成AI | **Buy** | SaaS活用で即時展開。開発コスト最小化 | Kofax / UiPath Document AI |
| 音声・画像認識（倉庫内） | **Buy** | 汎用基盤モデル流用で品質担保 | Azure Cognitive Services |

**[推奨] ベンダーロックイン回避原則**: コア差別化領域（需要予測・冷凍倉庫AI）は自社所有のSnowpark MLで構築し、モデルアーティファクトの可搬性を確保する。汎用領域のBuyは標準API経由に限定し、独自SDKへの依存を排除する。

### SAP × Snowflake × Power BI 統合設計

#### (1) SAPからSnowflakeへのリアルタイムデータ連携手法

| 連携方式 | 対象データ | レイテンシ | 推奨 | 採用優先度 |
|:---:|:---:|:---:|:---:|:---:|
| **SAP Datasphere + Snowflake Native Connector** | FI/CO/MM/SD全般 | 準リアルタイム（15分）| [推奨] | 第1優先 |
| **CDC（Debezium on SAP HANA）** | トランザクション系（受注・在庫）| ほぼリアルタイム（1-5分）| [推奨] | 第2優先 |
| **SAP Direct Extractor（DataSources）** | 標準レポートデータ | 日次バッチ | 既存踏襲 | 補完的利用 |
| **JDBC/ODBC経由Talend ETL** | 非標準テーブル | 日次バッチ | 非推奨 | 廃止方向 |

**[推奨] 移行シーケンス**: 初期はSAP Direct Extractor（日次）でSnowflakeへのデータ移行を進め、稼働確認後にCDCを追加し日次→準リアルタイム化を実現する。SAP Datasphereとのネイティブ連携は2027年度Q3以降に適用する。

#### (2) 管理会計データの直接抽出アーキテクチャ（月次→日次化の技術設計）

**現状の問題**: SAP CO（管理会計）モジュールのコストセンタ実績・製品別原価は月次締め後のみDIVAへ連携。経営管理部門の意思決定が最大30日遅延。

**技術解決策:**

```
【日次化技術フロー】

SAP CO（仮締めデータ）
  └─ CDC（Debezium）→ Kafka → Snowpipe → Snowflake Silver層
  └─ 仮締め実績 + 予算データ = 「試算レポート（日次版）」をPower BIで公開

SAP FI（確定仕訳）
  └─ SAP Datasphere → Snowflake Bronze層（15分間隔）
  └─ 確定後データは自動的に日次レポートを上書き更新

DIVA（月次確定値）
  └─ 月次バッチ（現行維持）→ Snowflake Gold層の「確定区分」フラグで管理
  └─ 仮値・確定値をSnowflakeでバージョン管理（SCD Type 2）
```

**期待効果**: 管理会計の日次モニタリングにより、月次比較から週次トレンド分析へ移行。経営層の意思決定サイクルを30日→7日に短縮（意思決定速度向上効果: 推定経営コスト削減3.5億円/年）。

#### (3) Power BI セマンティックレイヤー設計（グループ共通データモデル）

**現状**: グループ3社（ニチレイフーズ・ニチレイロジ・持株）でKPI定義が異なる（例: 「売上」の計上タイミング相違、「在庫日数」の計算式が2種類存在）。

**[推奨] グループ統合セマンティックレイヤー（Power BI Premium + Snowflake）:**

| データドメイン | 共通KPI定義 | Snowflake Gold層テーブル | Power BIメジャー名 |
|:---:|---|---|---|
| 売上・収益 | 請求確定日基準（DIVA確定値） | `fact_revenue_group` | `[売上高_確定]` / `[売上高_仮]` |
| 在庫 | 在庫日数=月末在庫/日次出庫平均 | `fact_inventory_daily` | `[在庫日数_標準]` |
| 物流コスト | 輸配送費+保管費（全倉庫合算） | `fact_logistics_cost` | `[物流コスト_全社]` |
| 品質 | クレーム率=クレーム件数/出荷件数 | `fact_quality_claim` | `[クレーム率_月次]` |
| 需要予測精度 | MAPE（Mean Absolute Percentage Error）| `fact_forecast_accuracy` | `[予測精度_MAPE]` |

**移行計画**: MotionBoard（グループ会社A）は2027年度末廃止。TableauはデータソースをSnowflake統合後に維持または廃止を選択（ライセンスコスト比較後判断）。

```
  └─ CVEスキャン結果・修正済みフラグ
       │
       ▼
定期スキャン（週次）
  └─ 新規CVE発生時: 72時間以内にパッチ適用またはモデル停止
  └─ 高リスクCVE（CVSS 9.0+）: 即時停止・インシデント対応
```

**[推奨]** 日立F&Lとのベンダー契約にAI-SBOM提出義務（四半期）とCVE対応SLA（重大: 24h / 高: 72h）を明記する。違反時のペナルティ条項を契約に含める。

#### OSSモデルの脆弱性スキャン（Snowflake上のMLモデル）

- Snowpark Container Services内のDockerイメージは週次でAqua SecurityによるCVEスキャンを実施
- スキャン結果をSnowflakeのセキュリティダッシュボード（Power BI連携）で可視化
- CVSS 7.0以上の脆弱性はAzure Security Centerへ自動通知

### 領域 #5: プライバシー保護技術（PETs）

**背景**: ニチレイグループは消費者向け冷凍食品の購買データ・グループ各社の営業秘密（製造コスト・レシピ等）を保有する。これらをAI学習に活用する際の個人情報保護法・改正個人情報保護法への対応が必須。

#### 差分プライバシー（DP）: 需要予測での顧客購買パターン保護

| 適用場面 | DP実装方式 | ε（プライバシー予算）| 精度影響 | 採用判定 |
|---|:---:|:---:|:---:|:---:|
| 消費者購買データ集計（POS連携）| Laplacianノイズ付加 | ε=1.0 | MAPE悪化約3% | **[採用]** |
| 地域別販売傾向レポート | Gaussian Mechanism | ε=2.0 | 誤差±5% | **[採用]** |
| 個別消費者の購買履歴 | 局所的差分プライバシー（LDP）| ε=0.5 | 精度大幅低下 | [保留: 要再評価] |
| グループ内従業員データ | 対象外（個人情報保護法に基づく通常処理）| ─ | ─ | ─ |

**[推奨]** POS連携データ（外部小売データ）にε=1.0のDP適用を2026年度Q3までに実装。MAPE精度の3%悪化は許容範囲内であり、法的コンプライアンスリスクの回避を優先する。

#### 連合学習（FL）: グループ各社データを共有せずグローバルモデル構築

**問題**: ニチレイフーズ（製造）とニチレイロジ（物流）は別法人であり、データの直接共有には法的・競合リスクが存在する。

```
連合学習アーキテクチャ

【ニチレイフーズサイト】             【ニチレイロジサイト】
  ローカルモデル学習                  ローカルモデル学習
  (製造実績データ)                   (物流コストデータ)
       │ 勾配のみ送信                      │ 勾配のみ送信
       └──────────────┬──────────────────┘
                      ▼
              【持株会社: Flower FL Server（Azure）】
                      グローバルモデル集約（FedAvg）
                      モデル配布 → 各社ローカル更新
```

**適用モデル**: 需要予測モデル（生産計画×物流コスト最適化の統合モデル）
**期待効果**: グループ統合需要予測精度 +8% vs. 個社単独予測（推定）
**実装ツール**: Flower（フレームワーク）+ Snowflake共有ストレージ（グローバルモデルのみ保存）

#### 食品安全データの秘密計算検討

| 技術 | 適用候補 | 実現可能性 | 優先度 |
|:---:|---|:---:|:---:|
| 準同型暗号（HE） | 原材料配合比のAI学習（レシピ秘匿）| 低（計算コスト×100倍）| 2028年以降 |
| Secure MPC | 競合他社との業界ベンチマーキング | 中（PoC実績あり）| 2027年PoC |
| TEE（Intel SGX）| 食品安全検査データの匿名化処理 | 高（クラウド対応済み）| 2027年採用 |

**[推奨]** 秘密計算は現時点ではTEE（Azure Confidential Computing）のみ実用フェーズに入っており、食品検査データの処理から適用を開始する。準同型暗号は技術成熟度が低いため2028年以降の再評価とする。

### 領域 #9: Zero Trust for AI

#### Snowflakeへのゼロトラストアクセス設計

**設計原則**: 「信頼しない、常に検証する（Never Trust, Always Verify）」をSnowflakeへの全アクセスに適用。

```
Zero Trust アクセスフロー（Snowflake）

ユーザー/アプリ
  │
  ▼ ①認証（Azure AD / Entra ID MFA必須）
Azure AD Conditional Access
  │ IP制限 + デバイスコンプライアンスチェック + リスクベース認証
  ▼ ②認可（RBAC + Row-Level Security）
Snowflake
  ├─ RBAC: ロール別アクセス権（最小権限原則）
  │   ├─ `ROLE_AI_READONLY`: AI推論用読み取り専用
  │   ├─ `ROLE_ETL_WRITER`: ETLジョブ専用書き込み
  │   ├─ `ROLE_ANALYST_FINANCE`: 財務データのみ参照可
  │   └─ `ROLE_ADMIN`: DBAのみ（多段承認必須）
  ├─ Row-Level Security（RLS）: グループ会社データを法人コードでフィルタ
  ├─ Column-Level Security（CLS）: 個人情報列はマスキングポリシー適用
  └─ Network Policy: 承認済みIPレンジのみ（VPN/PrivateLink必須）
       │
       ▼ ③監視（SIEM統合）
Snowflake Access History → Azure Sentinel
  └─ 異常アクセス検知: 深夜アクセス・大量データ抽出を自動アラート
```
